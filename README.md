5. **미션 진행 원칙**

   > `빠르게 구현하고, 빠르게 PR을 보내라.`
   코드 리뷰는 가장 효과적인 학습 수단 중 하나다.
   PR을 빨리 보낼수록 피드백을 빨리 받고, 더 많이 배운다.
   >

   | 이렇게 하지 않는다 | 이렇게 한다 |
       | --- | --- |
   | 완벽하게 만들고 PR을 보낸다 | 동작하는 코드가 되면 바로 PR을 보낸다 |
   | 부족한 부분을 숨긴다 | 부족한 부분을 인식하고 공유한다 |
   | 리뷰가 올 때까지 기다린다 | PR을 보내고 다음 단계를 준비한다 |

   완벽한 코드가 아니라 **피드백을 받을 수 있는 코드**가 목표다.
   PR이 빠를수록 리뷰어와의 대화가 빨라지고, 피드백에서 더 많이 배운다.

6. **완료의 기준**


| 이것은 완료가 아니다 | 이것이 완료다 |
| --- | --- |
| 코드가 돌아간다 | "왜 이렇게 했는지" 설명할 수 있다 |
| 테스트가 통과한다 | 처음과 달라진 생각을 말할 수 있다 |
| 시간 내에 제출했다 | 다음에 뭘 학습할지 스스로 정할 수 있다 |

> "제출했다"는 시작이다. "달라졌다"가 끝이다.
>

7. **레벨1을 마치면**

   ### **할 수 있어야 하는 것**

    - 문제를 자신의 언어로 정의할 수 있다
    - 설계 결정의 근거를 설명할 수 있다
    - 규칙을 만들고, 적용하고, 수정할 수 있다
    - 처음과 달라진 생각을 말할 수 있다
    - 다음 학습 방향을 스스로 설정할 수 있다

   ### **만들게 될 규칙 카테고리**

    ```
    테스트: 단위 기준, 제외 기준, 어려울 때 행동
    설계: 책임 분리, 상태 소유, 불변/캡슐화
    다형성: 조건문 대체 기준, 인터페이스 설계
    영속화: 도메인/저장소 분리, 저장 경계
    ```
