## 인터넷이란?

> 2대의 컴퓨터를 연결한다고 생각해보자. 어떻게 하면 될까?

2대의 컴퓨터는 케이블 하나로 직접 연결하면 된다. 간단하다.

그런데 **10대**라면? 각 컴퓨터를 나머지 9대에 모두 직접 연결하면 총 **45개의 케이블**이 필요하다. **100대**라면? **4,950개**. 이런 방식은 현실적이지 않다.

### 네트워크 토폴로지 (Network Topology)

이 문제를 해결하기 위해 다양한 연결 구조, 즉 토폴로지(topology, 네트워크에서 컴퓨터들이 어떤 형태로 배치되고 연결되는지를 나타내는 구조)가 등장했다.

* **버스(Bus)** - 하나의 중앙 케이블에 모든 컴퓨터가 연결됨
* **스타(Star)** - 중앙의 허브/스위치에 모든 컴퓨터가 연결됨. 집/사무실 같은 소규모 네트워크(LAN)에서 가장 흔한 형태.
* **링(Ring)** - 컴퓨터들이 원형으로 연결됨.
* **메시(Mesh)** - 컴퓨터들이 서로 다수의 경로로 연결됨. 하나의 경로가 끊겨도 우회 가능. 인터넷 전체의 기본 구조.
  ![Pasted image 20260314200309.png](https://techcourse-storage.s3.ap-northeast-2.amazonaws.com/51a44613bfda4d87921923adda344b0f)

---

> 그렇다면 이 네트워크들끼리는 어떻게 연결할까?

각 네트워크의 중심에 **라우터(Router)** 를 두고, 라우터끼리 연결하면 된다. 이렇게 **네트워크의 네트워크**가 바로 **인터넷(Internet)** 이다.

역사적으로 **ARPANET**이 인터넷의 최초 연결 지점들을 서로 이어주었다.

### 라우터 (Routers)

> 우체국이 편지의 주소를 보고 다음 우체국으로 전달하듯, 라우터는 데이터의 목적지 IP를 보고 다음 라우터로 전달한다.

* 데이터의 목적지 IP 주소를 보고 **다음 라우터로 전달**하는 장치
* 데이터는 A에서 B로 가는 **여러 경로**를 가질 수 있다. 하나의 라우터가 혼잡하면 다른 경로를 통해 데이터가 흐를 수 있다.
* 데이터들은 서로 **다른 경로**로 전달될 수도 있고, 도착 후 순서대로 재조립된다.

### 패킷 (Packets)

데이터는 인터넷을 통해 한 번에 통째로 전송되지 않는다. **패킷**이라는 작은 조각으로 나뉘어 전송된다.

> 큰 편지를 하나의 봉투에 넣으면 너무 크다. 여러 장으로 나눠서 각각 봉투에 넣고, 봉투마다 "1/4", "2/4"처럼 번호를 매겨 보낸다.

* 각 패킷에는 **발신 IP 주소**, **수신 IP 주소**, **순서 번호**가 적혀 있다.

### TCP/IP

**TCP/IP**는 컴퓨터가 인터넷을 통해 데이터를 전송할 수 있게 해주는 두 가지 프로토콜이다. 각각 하는 일이 다르다.

#### IP (Internet Protocol) - "어디로 보낼까?"

* 인터넷에서 컴퓨터가 서로를 **식별**하고, 패킷을 목적지까지 **라우팅을 가능하게** 하는 역할
* 모든 컴퓨터는 세계에서 고유한 주소를 가진다.
* 주소 형식: `#.#.#.#` (각 숫자는 0~255)
* IP 주소는 **32비트**이며, 약 **40억 개** 이상의 주소를 수용할 수 있다. (IPv4)
* 주소가 부족해지면서 **IPv6** (128비트)로 전환 중

![Pasted image 20260314210849.png](https://techcourse-storage.s3.ap-northeast-2.amazonaws.com/83765a194d4e4f45ac330d65510da0db)

> 공인 IP vs 사설 IP
> * **공인 IP**: 인터넷에서 고유한 주소. **ISP(Internet Service Provider, KT·SKT·LG U+ 같은 인터넷 서비스 제공자)** 가 가정의 라우터에 할당한다.
> * **사설 IP**: 집이나 회사 같은 **내부 네트워크에서만** 쓰이는 주소. `192.168.#.#`, `10.#.#.#` 등
> * `192.168.0.5` 같은 주소가 보인다면 그것은 사설 IP이다. 공유기(라우터)가 DHCP로 자동 할당해준 것.

---

#### TCP (Transmission Control Protocol) - "제대로 도착했나?"

* TCP의 핵심 역할은 데이터가 **빠짐없이, 순서대로** 도착하도록 보장하는 것이다.
    * **시퀀스 번호**로 패킷의 순서를 추적한다.
    * 패킷이 손실되면 누락된 패킷을 다시 요청한다.
    * 모든 데이터가 전송되고 수신되었음을 **확인(acknowledge)** 한다.
* **포트 번호(port numbers)** 를 통해 어떤 서비스인지 구분한다.
    * `80` → HTTP
    * `443` → HTTPS
    * `25` → 이메일(SMTP)

![Pasted image 20260314211552.png](https://techcourse-storage.s3.ap-northeast-2.amazonaws.com/475e29cb48a7429997a5051f6b15d2b1)

> 참고: UDP (User Datagram Protocol)
> TCP와 달리 패킷 도착을 **보장하지 않는** 프로토콜이다. 대신 **빠르다**.
> 화상회의(Zoom), 실시간 스트리밍처럼 패킷 하나를 잃더라도 멈추지 않고 계속 흘러가야 하는 경우에 사용된다.

### HTTP / HTTPS

**HTTP**(HyperText Transfer Protocol)는 웹 브라우저와 서버가 통신하기 위한 **애플리케이션 레벨 프로토콜**이다. **TCP** 위에서 동작하므로, 데이터의 신뢰성 있는 전송이 보장된다.

* `https://www.example.com` 주소는 사실 끝에 `/`가 암묵적으로 포함되어 있다.
* `/` 뒤에 오는 것이 **경로(path)** 이다.
    * 예: `https://www.example.com/folder/file.html`
* HTTP는 대표적으로 **GET**(데이터 요청)과 **POST**(데이터 전송) 요청을 사용한다.
* **HTTPS**는 HTTP에 **암호화(TLS/SSL)** 를 추가한 것이다. 데이터가 중간에 도청되어도 내용을 알 수 없다.

### DNS (Domain Name System)

* 웹사이트를 방문할 때마다 `142.250.207.100` 같은 숫자를 기억하는 것은 현실적이지 않다.
* DNS는 `google.com` 같은 도메인 이름을 IP 주소로 변환해주는 **서버들의 집합**이다.

> DNS 조회 과정:
> 클라이언트는 **리졸버**에만 질문하고, 리졸버가 아래 서버들을 단계적으로 찾아간다.
>
> * **Local/ISP DNS 리졸버(Resolver)**: 모든 질문을 대신 처리해주는 서버. 캐시가 없으면 아래 서버들에 직접 질문한다.
> * **루트 네임 서버(Root Name Server)**: 전 세계 13개(논리적). "`.com`은 TLD 서버에 물어봐요"라고 안내한다.
> * **TLD 네임 서버(TLD Name Server)**: `.com`, `.net`, `.kr` 같은 도메인을 담당. 권한 서버를 안내한다.
> * **권한 네임 서버(Authoritative Name Server)**: 해당 도메인의 **실제 IP 주소를 보유**한 서버.

## 구글에 "고양이"를 검색하면 무슨 일이 일어날까?

위에서 배운 개념들이 실제로 어떻게 동작하는지, 하나의 흐름으로 따라가보자.

**1. DNS로 IP 주소 찾기**

* 브라우저가 **DNS 리졸버**에 물어본다: "[www.google.com](https://www.google.com)의 IP 주소가 뭐야?"
* 리졸버가 루트 → TLD → 권한 네임 서버를 단계적으로 조회한다.
* 최종 응답: `142.250.207.100`

**2. TCP/IP로 연결**

* 내 컴퓨터(`192.168.0.5` ← 사설 IP)에서 구글 서버(`142.250.207.100` ← 공인 IP)로 **TCP 연결**을 시작한다.
* HTTPS이므로 **포트 443**을 사용한다.
* 내 컴퓨터 → 공유기(라우터) → ISP → 여러 라우터를 거쳐 → 구글 서버에 도착

**3. HTTP 요청 전송**

* 브라우저가 구글 서버에 **GET** 요청을 보낸다:

```http
GET /search?q=고양이 HTTP/1.1
Host: www.google.com
```

**4. 구글 서버가 응답**

* 구글 서버가 검색 결과를 담은 HTML을 돌려보낸다:

```http
HTTP/1.1 200 OK
Content-Type: text/html
```

* 이 응답 데이터는 여러 **패킷**으로 쪼개져서 전송된다.
* 각 패킷은 **라우터**를 통해 서로 **다른 경로**로 전달될 수 있다.
* **TCP**가 모든 패킷이 빠짐없이, 순서대로 도착했는지 확인한다.

**5. 브라우저가 화면에 표시**

* 받은 HTML을 해석해서 고양이 검색 결과를 화면에 보여준다.

> 정리
> **DNS**로 주소를 찾고 → **TCP/IP**로 연결하고 → **HTTP**로 요청/응답을 주고받는다.
> 위에서 배운 모든 개념이 하나의 검색 동작 안에 들어있다.

여기서 3번부터 5번까지, 즉 **HTTP로 요청/응답을 주고받고 HTML로 화면을 그리는 부분**이 바로 **웹**의 영역이다. 반면 1~2번의 DNS, TCP/IP는 웹보다 하위 계층인 **인터넷(네트워크) 영역**이다.

---

## **인터넷 != 웹**

* **인터넷** = 컴퓨터들이 연결된 물리적 네트워크 인프라 (케이블, 라우터, TCP/IP 등). 데이터가 오갈 수 있는 **도로망**.
* **웹** = 인터넷 위에서 동작하는 **서비스 중 하나**. HTML 문서를 HTTP로 주고받는 것.

인터넷이라는 도로망 위에는 웹 말고도 다양한 서비스가 함께 달린다.

* 이메일 (SMTP)
* 파일 전송 (FTP)
* 온라인 게임
* 메신저

> 인터넷이 **도로망**이라면, 웹은 그 도로를 이용하는 **택배 서비스** 중 하나다. 도로 위에는 택배 말고도 버스(이메일), 택시(메신저) 등 다양한 서비스가 함께 달린다.
