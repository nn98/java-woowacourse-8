# `4. 피드백 활동`

피드백 활동

캐리: 2번 잘 작동. 1번은 추상적이다 보니 적용하기 어려움
시간/테마/예약에 대해 각각이 독립적인 리소스인지 여부에 따라서인데
그런 내용이 정해져있진 않아서 규칙을 적용하기 힘들었고,
단순한 리소스의 정의는 내 관점으로 바로바로 가능했지만
복잡한 리소스, 여러 스코프로 요청하는 엔드포인트를 설계하는 데엔 규칙이랄게 애매

기본적인 / 단순한 엔드포인트는 심플하고 의견의 상충이 없지만
복합적인 엔드포인트 설계 기준에 대한 규칙이 있는것이 좋지 않을까
설계한 방식은 /theme/time?date= 구조.
변경 가능성에 따라 리소스를 식별하고 path 에 활용하는 것이?

캐리의 결론 : 리소스의 독립성/종속성이 API 설계를 결정하는 기준

대길 리뷰어 : 고유 식별자만이 리소스이기에 path 로, 나머지는 parameter 로.

좋은 API

대길 : 명확하고 직관적인 API. 뚜렷하게 알아볼 수 있는 API
캐리 : 응답을 기대할 수 있고 일관된 규칙을 가진 API
마이찬 : 이해하기 쉬운 API. 어떻게 작성하는지보단 그 결과가 명확히 이해하기 쉬운지
어떻게, 행위는 숨기고 무엇을 하는지를 드러낸다
글렌 : 엔드포인트는 무엇을, 메서드를 어떻게 를 표현하고 상호 보완하고 상호 필수적인 API

대길 : 명확성
캐리 : 일관성
마이찬 : 은닉성
글렌 : 상호보완성
