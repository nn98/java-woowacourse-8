### 6. 다형성으로 대체할 수 있는 조건문의 기준은?

- 다형성으로 대체할 수 있는 조건문은 객체의 종류에 따라 다른 동작을 수행하기 위해 사용되는 조건문이라고 생각한다. 예를 들어 기물의 종류(차, 마, 상, 사, 장, 포, 졸)에 따라 이동 규칙이 달라지므로, 이를 조건문으로 분기하면 기물 종류가 늘어날수록 코드가 복잡해지고 수정 범위가 넓어질 수 있다.
- 이러한 경우 각 기물 객체가 자신의 이동 규칙을 직접 구현하도록 설계하면, 기물의 종류를 조건문으로 비교할 필요가 없어진다. 즉, 이동 가능 여부를 판단하는 책임을 기물 객체에게 맡기고, 공통 인터페이스나 추상 클래스를 통해 동일한 메시지를 사용할 수 있도록 하면 다형성을 활용할 수 있다.
- 반면 보드 범위를 벗어나는지 확인하거나, 같은 팀 기물이 있는지 검사하는 것과 같은 조건은 객체의 종류와 관계없이 공통적으로 필요한 검증이므로 다형성으로 대체하기 어렵다. 이러한 조건문은 여전히 필요한 로직이라고 생각한다.
- 따라서 다형성으로 대체할 수 있는 조건문의 기준은 객체의 종류에 따라 동작 방식이 달라지는 경우이며, 이러한 분기 로직을 객체 내부로 이동시켜 오버라이딩을 통해 해결할 수 있다고 생각한다.

고래 : 객체들이 동일한 메시지를 받았을 때 서로 다르게 행동해야 할 때

### 8. 공통 인터페이스는 어떻게 설계하는가?

- 공통 인터페이스는 여러 객체가 같은 행위를 공통적으로 수행해야 하지만, 그 구체적인 방식은 서로 다를 때 설계한다고 생각한다.
- 장기의 경우 모든 기물은 공통적으로 “이동 가능 여부를 판단하는 행위”가 필요하다. 그러나 차, 마, 상, 사, 장, 포, 졸은 각각 이동 규칙이 다르므로, 같은 행위를 하더라도 내부 동작은 서로 달라야 한다.
- 이때 공통 인터페이스를 두면, 모든 기물이 동일한 방식으로 메시지를 받을 수 있다. 예를 들어 모든 기물이 canMove(from, to, board)와 같은 메서드를 가지도록 인터페이스를 정의하면, 호출하는 쪽에서는 기물의 종류를 구분하지 않고도 동일한 방식으로 이동 가능 여부를 물어볼 수 있다. 실제 판단 로직은 각 기물 클래스가 자신의 규칙에 맞게 구현하면 된다.
- 즉, 공통 인터페이스는 “무엇을 할 수 있어야 하는가”를 통일하는 역할을 하고, 각 구현 클래스는 “어떻게 할 것인가”를 담당한다.
- 이러한 구조를 사용하면 기물 종류를 조건문으로 분기하지 않아도 되고, 책임이 각 객체에 자연스럽게 분리되며, 새로운 기물이 추가되더라도 기존 코드를 크게 수정하지 않고 확장할 수 있다고 생각한다.

고래 : 공통 인터페이스는 “무엇을 할 수 있어야 하는가”를 통일하는 역할을 하고, 각 구현 클래스는 “어떻게 할 것인가”를 담당한다.
