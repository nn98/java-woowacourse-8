<h2>토론 활동 규칙</h2>
<ul>
<li>사전학습에서 설계한 <strong>API와 그 이유</strong>를 근거로 사용한다</li>
<li>"~가 맞는 것 같다"가 아니라 "~로 결정했다 / ~을 그렇게 봤다"로 말한다</li>
<li>의견 충돌은 좋은 것, 성급하게 합의하지 않는다</li>
<li>"RESTful 규칙에 따르면..."이 아니라 <strong>왜 그렇게 하는 게 좋다고 생각하는가?</strong> 로 파고든다</li>
</ul>

<h2>토론 활동 결과물</h2>
<p>다음 4가지 규칙을 만든다 — 각 토론 주제에 대응한다.</p>
<ol>
<li><strong>리소스 식별 기준</strong> - URL을 정할 때 무엇을 리소스로 보고, 무엇을 필터·표현 조건으로 보는가</li>

<li><strong>관리자/사용자 API 분리 기준</strong> - 같은 자원을 다루는 관리자와 사용자 API를 분리하는가, 합치는가</li>
<li><strong>우리 그룹의 "좋은 API" 정의</strong> - 동작하는 것을 넘어, 어떤 조건을 만족해야 좋은 API라고 부를 것인가</li>
</ol>

# `2. 토론`

주제 1: API를 설계할 때
가장 먼저 결정해야 하는 것은 무엇인가?

* 글렌 : 도메인에 대한 정의
* 마이찬 : API 엔드포인트 / 메서드 종류 / 상태 코드 등을 고려했을 때
    * 엔드포인트 리소스 설정과 그 근거
    * 리소스의 순서 / 계층 구조와 흐름
    * 캐리의 질문
        * 엔드포인트 설정을 위해 리소스 식별 필요
        * 엔드포인트의 정의는 행동의 정리까지 완성되었다는 뜻
          HTTP 메서드와 그 내부 로직까지
        * 그렇다면 엔드포인트의 정의에 선결되는 일이지 않는가
        * 엔드포인트라는 용어 자체가 가장 먼저 결정할 수 있는지
* 등에 대한 본인의 근거,
    * 큰 틀을 짜서 엔드포인트, 메서드 종류, 상태 코드의
      API 명세를 전반적으로 작성하는 것
* 대길 : 도메인에 대한 설계
    * 기존 : 별다른 생각 없이 API 설계
    * 엔드포인트 / 쿼리파라미터 등을 고려해 보았을 때,
      리소스란 용어가 모호.
    * 그 결정을 위해 도메인에 대한 설계가 명확해야.
* 캐리 : 요청에 집중한다.
    * 요청에 포함된 내용(명사) 를 기준으로 리소스를 추출
    * 리소스를 기반으로 s 붙여서 엔드포인트 설정

주제 2: 이미 예약된 시간을
서버가 걸러야 하는가,
클라이언트가 걸러야 하는가?

* 마이찬 : 예약 가능 시간들 싹 가져오면 효율적.
* 마이찬 : 예약 가능 시간ㄷ시간들 싹 가져오면 효율적.
* 데이터나 정보에 대한 책임이 서버이기에
* 대길 : 서버가 관리
* 캐리 : 서버가 관리해야 한다.
    * 서버가 데이터를 관리하고 조작하는 상황인데,
      데이터의 정합성을 관리하는 것은 서버의 책임이다.
    * 데이의 관리 책임은 곧
      데이터의 무결성과 정합성을 관리하는 책임과 같기에.
    * 클라이언트가 관리한다는 것이 어불성설
* 마이찬 : 서버가 걸러야
    * 주제의 근본 목적을 생각했을 때,
    * 예약 시간에 대한 관리같은 사전적인 정보가 있어야 사용자가 예약이란 기능을 수행할 수 있다.
    * 서버가 거르는 방식은 어떻게 구현되고
    * 예약가능 시간을 가져올 것인가
    * 클라이언트가 거른다면 테마에 대한 정보를 다 가져오고
      그 정보를 걸러내서
* 글렌 : 둘 다 걸러내야 한다
    * 캐리의 궁금점
        * 사전 학습 과정에서 예약 가능 시간만 보내줄지
          예약 불가능한 시간도 보내줄지 고민해 보았는가?
        * 요구사항을 분석했을 땐 예약 가능한 시간만 보내준다 생각.
          그것이 맞나? 다 보내줘야 한다 생각하진 않나?
        * 대길 : 찜꽁을 기준으로 생각했을 때,
          예약 가능한 시간만 활성화되게 뜨는데
          백엔드에선 예약 가능한 시간만 보내 주는 것이 맞다
          프론트엔드에서 내려오지 않은 시간은 알아서 처리
        * 마이찬 : 예약 불가한 데이터는 프론트가 알아서 처리하는 책임이지 않나
        * 글렌 : 활성 비활성 다받아와야 불필요한 추가 조회 없이
          한 번에 프론트가 화면을 그려줄 수 있지 않을까
        * 요구사항과는 별개로 모든 시간과 예약 가능 여부를 받아오는 별도의 엔드포인트를 두겠다
        * 마이찬 : 위 내용에 동의. 예약 가능 시간 단위가 있을 수도 있는데,
        * 위와 같은 궁금증에 대한 결과로, 서버가 걸러내는 것이 맞지 않나.

주제 3: 관리자 API와
사용자 API 는 분리해야 하는가?

* 글렌 : 분리해야 한다.
    * 기존 관리자 API 를 변경하는 한이 있더라도
    * 도메인의 측면에서 무엇이 핵심적인가 무엇이 우선인가 생각
* 대길 : 분리해야 한다.
    * 일관성(reservation) 은 유지하되
    * 캐리 : 사용자 관리자 분리하지 않아야.
      권한 설정으로 분리하는 것이 맞지
      엔드포인트 자체가 예약이라는 리소스를 표현하는데
      예약 엔드포인트는 예약 그 자체에 관련된 내용을
      일관성 있게 관리해야 한다 생각.
      그것이 더 좋은 생각이 아닌가?
* 마이찬 : 분리해야 한다.
    * 분리의 기준
        * 관리자 / 사용자의 응답 데이터 구조가 상이할 수도
        * 관리자 / 사용자의 인증의 내용이 상이할 수도
        * 공통적으로 처리할 부분은 공통으로 추출하고
          그렇지 않은 부분을 분리해야 한다
    * 분리의 방법
        * 대길과 유사하게 앞에 용어를 추가해서 분리하고
          일관성을 유지하지 않을지
    * 공통적인 부분
        * 조회와 같이 public 한 내용들은 추출된 공통 엔드포인트
    * 캐리의 의문
        * 권한에 대한 관리가 필연적일 텐데
        * 그렇다면 권한을 기준으로 분리 아닌 구분이 가능한데
        * 이미 구분/분리가 될 수밖에 없는 상황에
          엔드포인트를 굳이 또 분리할 이유가 있는가??
    * 대길의 답변
        * 잘 모르는 내용이지만
        * 두 번의 필터링에 대한 리뷰에서 경험한 바에 따르면
        * 더 방어적으로 관리하기 위해서라도 분리하고 관리해야
    * 마이찬의 답변
        * 하나의 엔드포인트에서 권한에 따라 분기하는 소요 존재
        * API 분기를 통해 더
    * 캐리의 반론
        * 같은 리소스라 같은 동작할텐데 그런 안티패턴?
        * 마이찬 : 지금은 그래도 관리자의 요청이 별도의 로직을
          필요로 한다면 그 때의 분기는 달라진다
        * 예약이라는 리소스는 같다. 그럴 때 다른 엔드포인트를
          사용한다면 그게 직관성을 저해하는 엔드포인트이지 않나?
        * 같은 조회를 수행하는 부분인데 분리되지 않고
          동일한 것이 더 자연스럽고 일관적이지 않나
        * 수정삭제는 메서드 차이로 구분하고 정의하고
          거기에 Auth 를 씌우는 것이 엔드포인트를 늘이는 것 보다 더 효율적이고 규칙에 적합하지 않나
        * 마이찬 : 애초 API 란 클라이언트 입장에서 필요한 데이터를 조회하는 것.
            * 사용자용 / 관리자용 조회한 데이터의 종류가 다르다면?
            * 관리자용/사용자용 API 의 데이터가 달라질 경우엔 아예 별도의 조회를 하는 것이

주제 4: “좋은 API"와
"동작하는 API"는 무엇이 다른가?

* 좋은 API : API 명세를 보고 요청과 행동을 직관적으로 식별할 수 있는 것
* 좋은 API : 가독성이 뛰어난 API.
* 캐리 : 좋은 API : 가독성.
    * URL 과 메서드만으로 그 내용을 추론할 수 있어야.
    * 방금 얘기했듯이 그 가독성의 기준이 위와 같다.
    * 동작하는 API : 그 외의 모두
        * 산출물에서도 좋은 API 정의니
        * 리소스의 정의와 메서드 시그니처 등의 컨벤션으로
          좋은 가독성을 보유한 것이 좋은 API. 마이찬 : API 명세를 보고 요청과 행동을 직관적으로 식별할 수 있는 것
    * 동작하는 API : 내부 동작의 구현을 직관적으로 식별하기 힘든 API
* 대길 : 동작하는 API 가 좋은 API 가 아니라는 뜻은 아니다
    * 지금까지의 내 API 는 동작하는 API 에 가깝다
    * API 가 호출과 동작만 보장하면 되는 것이라 생각했기에
    * 명확한 기준을 가지고 작성된 API 가 좋은 API 가 아닌가
    * 마이찬 : 대비/반대의 개념이 아닌 포함관계
* 글렌 : 이 하 동 문
    * 동작하는 API 중 좋은 API 가 존재하고,
    * 동작한다 는 말 그대로 요청의 응답을 반환한다는 것,
    * 좋다 는 그 API 의 명세만 보고도 동작을 유추할 수 있다는 것

리소스

* 독립적 으로 생성 조회 삭제되는 데이터, 리소스의 정의에 대해 고민하고
  (ex 고유한 ID 를 붙여 식별할 수 있다면 독립적인 리소스다)
* 엔드포인트를 구성하는 URL 에 리소스의 종속/독립에 따라 path/query parameter
  중 무엇으로 배치할 것인지 선택한다
* 캐리 : N:M 관계 등등에 대해 서버가 아닌 클라이언트 관점에서 생각해야 한다
  API 는 개발자/서버의 입장이 아닌 /클라이언트 입장에,
  즉 도메인 모델이 아닌 멘탈 모델 측면에서 생각해야 한다

REST API란

예약 가능 시간 관리 방법 2가지

예약 해당 조건이 필요하다.

* 예약 시간 단위 (1시간, 30분)
* 예약 시작 시간
* 예약 종료 시간
* 예약 여부


1. 테마별 예약 가능 시간 테이블 한 행씩 관리
   서버에서 예약가능한 시간을 관리한다고 생각한다면, 테마별 예약 가능 시간에 대한 테이블에서 모든 테마별 예약 시간에 대한 row를 모두 소유하고 있어야 한다.
   즉, 예약 시간 단위를 1시간이라고 치면, 한 테마당 하루에 최대 24개의 행이 생긴다. 만약 30일 정도가 지난다면, 한 테마당 하루에 24개행 X 30 = 720 행이 생긴다. 만약 테마가 10개 정도
   있다면, 1달 정도 지나면 720 X 10 = 7,200개 행이 생긴다.

* 이렇게 된다면 일정 기간별 지난 일정에 대해서는 백업을 하거나 삭제를 하거나, 미래에 대한 예약 가능시간 관리에 대한 기준 일을 또 설정해두어야 한다.
* 또한, 각 타입 슬롯 한개 마다 한개의 행이 생기기 때문에 하나씩 관리가 필요하다.
* DB에서 조회시 인덱스로 성능을 향상 시킬 수 있다.

타입 슬롯 (테마, 시작시간, 종료시간)

2. 테마별 예약 가능 시간 비트마스크로 관리
   예약 시간 단위를 30분 1시간 중에 하나라는 요구사항이 있다면
   0과 1의 48개 비트마스크로 관리한다.
   예약이 됬다면 1 예약이 안됐다면 0으로 관리한다.
   한 테마의 한 날짜에 대한 예약 가능 시간을 1개의 행으로 관리할 수 있다.
   조회가 빠르다.
   하지만 동시성에 대한 제어가 면밀히 필요하다.

관리자 API와 사용자 API가 일관적이었다.

* 두 API를 같은 엔드포인트로 합칠 것인가, 분리할 것인가? 무엇을 기준으로 결정했는가?

1. 리소스 식별 기준

URL을 정할 때 무엇을 리소스로 보고, 무엇을 필터·표현 조건으로 보는가

Rule 1. 도메인 분석을 통해, 실행 흐름에 나타나는 명사를 리소스로 식별한다.
리소스 식별 규칙 자체는 추상적으로 설정하되,
API 명세 설계에 대한 각자의 관점에 맞춰 판단하고

* 서버/도메인 모델 관점
* 클라이언트/멘탈 모델 관점

이후 피드백 활동에서 각자의 경험을 공유해 규칙을 구체화한다.

2. 서버/클라이언트 책임 기준

어떤 판단을 서버가 책임지고, 어떤 판단을 클라이언트가 책임지는가

* (If-Then) 만약 판단에 필요한 원천 데이터가 서버에만 존재한다면 → 해당 로직은 서버가 책임지고 결과값을 내려준다.
    * 예: "예약 가능 여부(available)"는 서버가 기존 예약 현황을 계산하여 응답에 포함한다.

* (If-Then) 만약 클라이언트가 받은 데이터를 가공 없이 화면에 그리는 역할이라면 → 서버는 클라이언트가 바로 사용할 수 있는 형태(UI friendly)로 응답을 설계한다.

* (우선순위) 책임 판단 순서:

    1. 데이터의 무결성을 보장해야 하는 곳이 어디인가? (보통 서버)
    2. 클라이언트가 중복된 계산 로직을 가질 필요가 있는가?

* (금지) 클라이언트에게 비즈니스 로직 판단을 떠넘기지 않는다.
    * 이유: 데이터 불일치 문제를 방지하고 클라이언트의 복잡도를 낮추기 위함이다.


3. 관리자/사용자 API 분리 기준

같은 자원을 다루는 관리자와 사용자 API를 분리하는가, 합치는가

* (If-Then) 만약 리소스의 도메인 모델과 생성 행위가 동일하다면 → 관리자와 사용자 구분 없이 같은 엔드포인트를 사용한다.
    * 예: POST /reservations (누가 생성하든 '예약 생성'이라는 행위는 동일함)
* (If-Then) 만약 사용자의 권한에 따라 응답 필드의 구성이나 노출 데이터가 현저히 다르다면 → 도메인 목적에 따라 API를 분리하거나 권한 로직으로 분기한다.
* (우선순위) 결정 순서:

    1. 리소스의 식별자가 동일한가?
    2. 조회/수정의 목적과 Context가 동일한가?


4. 우리 그룹의 "좋은 API" 정의

동작하는 것을 넘어, 어떤 조건을 만족해야 좋은 API라고 부를 것인가

좋은 API

* 정의: 동작을 완벽히 보장하는 것은 물론, 뛰어난 가독성을 바탕으로 명세만 보아도 요청과 행동을 직관적으로 식별할 수 있는 API.
* 특징:
    * 직관성: URL(리소스 명사)과 HTTP 메서드(행위 동사)의 조합만으로 해당 API가 어떤 자원에 대해 무슨 작업을 수행하는지 누구나 명확히 추론할 수 있습니다.
    * 명확성: 리소스의 정의, 메서드 시그니처 등에 대해 팀이 합의한 명확한 기준과 규칙을 엄격하게 따릅니다.
    * 문서화의 역할: 잘 짜인 API 명세 자체가 훌륭한 설명서 역할을 하여, 개발자 간의 소통 비용과 유지보수 비용을 크게 낮춥니다.
