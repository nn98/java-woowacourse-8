# 방탈출 예약 시스템 명세 v2

## 0. 문서 규약

- **GWT 명세(§4)**: 각 시나리오는 그대로 테스트 이름·본문으로 변환 가능해야 한다. 변환 불가능한 문장은 이 섹션에 둘 수 없다.
  - 시나리오가 곧 테스트명/본문 
- **미결정 항목(§1)**: `D-n`으로 식별. 각 항목은 질문 / 선택지 / 제안 기본값 / 채택 시 활성화되는 GWT를 갖는다. 결정 즉시 해당 GWT를 §4로 이동한다.
- **결정 기록(§7)**: `R-n`으로 식별. 근거가 아닌 **무효화 트리거**(이 결정을 버려야 하는 관측 조건)를 반드시 포함한다.
  - 이 결정을 포기하는 것이 효율적인 시점/트리거를 포함
- 미결정 항목에 의존하는 시나리오는 `[D-n 의존]`으로 표기한다.

## 1. 미결정 항목 (Decisions Required)

### D-1. 대기 → 예약 전환 방식
- **질문**: 자동 전환인가, 관리자 수동 승인인가.
- **선택지**: (a) 예약 취소 시 대기 1번 자동 승격 (b) 관리자가 승인/거절
- **제안 기본값**: (a) 자동 전환 — 관리자 개입 UI/플로우가 현재 범위에 없음.
- **채택 시 활성화 GWT**:
  ```
  시나리오: 예약_취소시_대기_1번이_자동으로_예약된다
  Given 슬롯 S에 예약 1건과 대기 [갑(1번), 을(2번)]이 존재한다
  When 해당 예약이 취소된다
  Then 갑의 대기가 예약으로 전환된다
  And 을의 순번이 1번이 된다
  ```

### Decision 1
- 선택: 자동 전환
- 방식: 예약 취소 시 대기 확인 후 대기 1번 자동 승격
- **활성화 GWT**:
  ```
  시나리오: 예약_취소시_대기_1번이_자동으로_예약된다
  Given 슬롯 S에 예약 1건과 대기 [갑(1번), 을(2번)]이 존재한다
  When 해당 예약이 취소된다
  Then 갑의 대기가 예약으로 전환된다
  And 을의 순번이 1번이 된다
  ```

### D-2. 순번 재정렬 의미론
- **질문**: 중간 순번 이탈 시 압축하는가, 번호를 유지하는가.
- **선택지**: (a) 압축 — 2번 이탈 시 3→2, 4→3 (b) 번호 유지 — 결번 허용, 상대 순서만 보장
- **제안 기본값**: (a) 압축 — 사용자에게 노출되는 순번이 "내 앞에 몇 명"을 의미해야 함.
- **채택 시 활성화 GWT**:
  ```
  시나리오: 중간_대기_취소시_후순위_순번이_압축된다
  Given 슬롯 S에 대기 [갑(1), 을(2), 병(3)]이 존재한다
  When 을이 대기를 취소한다
  Then 병의 순번이 2가 된다
  And 갑의 순번은 1로 유지된다
  ```

### Decision 2
- 선택: 방식 자체를 수정한다. 대기의 컬럼으로 TIMESTAMP created_at 을 넣고 오름차순으로 정렬된 순서에 마주처 순번을 계산 
- GWT: 
  - 슬롯에 대기가 존재한다
  - 슬롯에 걸린 대기가 삭제/추가되면
  - 그에 맞춰 순번이 재정렬된다 
  - 스키마에 저장된 데이터들에 update 하지 않는다 

### D-3. 인기 테마 동률 처리
- **질문**: 예약 수가 같은 테마의 순서는 무엇으로 결정하는가. 10위 경계에 동률이 걸치면.
- **선택지**: (a) 테마 ID 오름차순 보조 정렬, 정확히 10개 절단 (b) 동률 전체 포함(10개 초과 허용)
- **제안 기본값**: (a) — 응답 크기 고정이 클라이언트 계약에 유리.
- **채택 시 §4.8의 `[D-3 의존]` 시나리오가 확정된다.**

### Decision 3
- 선택: 예약은 생성 시간을 저장하지 않으므로 ID 오름차순. 

### D-4. 승격 vs 신규 예약 우선순위 정책
- **질문**: 예약 취소로 슬롯이 비는 순간, 대기 1번의 승격과 제3자의 신규 예약 요청이 경합하면 누가 이기는가.
- **선택지**: (a) 대기자 우선 — 대기가 존재하는 슬롯은 신규 예약 차단 (b) 선착순 — 먼저 커밋한 쪽이 승리
- **제안 기본값**: (a) — 대기 제도의 존재 이유와 정합. 단, "대기가 존재하는 슬롯" 판정 자체가 경합 대상이므로 §6-C2와 함께 설계해야 한다.

### Decision 4
- 선택: 트랜잭션으로 예약 취소와 대기 승격을 하나의 단위로 묶기. 트랜잭션 진행 중 해당 슬롯에 대한 제3자의 신규 예약을 방지  
- 무효화 트리거: 예약 취소가 대기 승격보다 중요한 비즈니스 요구사항으로 간주된다면 두 개의 트랜잭션으로 분리

### D-5. 유효하지 않은 입력의 완전 열거
- **질문**: "빈 이름, 잘못된 날짜 형식 *등*"의 *등*을 닫아야 한다.
- **확정 필요 목록**: 이름 최대 길이 / 허용 문자 집합 / 날짜 표현 형식과 허용 범위(과거 차단은 §4.4에서 별도 처리) / 썸네일 URL 형식 검증 수준 / 테마 설명 최대 길이
- **현재 확정분은 §4.9에 GWT로 존재한다. 이 목록이 닫히기 전까지 §4.9는 불완전하다.**

### Decision 5
- 선택: 이름 최대 길이는 10자, 문자는 공백 한자 영어만, 날짜는 YYYY-MM-DD 형식으로 리퀘스트바디가 검증. 각각에 대한 유효성 검증 메시지를 예외에서 명시해서 응답

### D-6. 에러 응답 본문 스키마와 "의미 있는 메시지" 판정 기준
- **질문**: 에러 본문 필드 구성(코드/메시지/상세), 메시지의 노출 수준(내부 정보 차단 기준).
- **제안 기본값**: `{ "code": string, "message": string }` — code는 클라이언트 분기용 안정 식별자, message는 사용자 표시용. 스택트레이스·SQL·클래스명 노출 금지.
- **"의미 있는 메시지"의 판정 함수**: 본문에 code가 존재하고, message가 D-6 금지 항목을 포함하지 않으면 통과.

### D-7. 예약이 없는 슬롯에 대한 대기 신청
- **질문**: 빈 슬롯에 대기를 신청하면 거부하는가, 예약으로 자동 전환하는가.
- **선택지**: (a) 거부 — "예약 가능한 슬롯입니다" 안내 (b) 예약으로 생성
- **제안 기본값**: (a) — 대기의 정의("이미 예약된 슬롯에 신청")와 정합. (b)는 클라이언트 상태가 낡았을 때 UX는 좋지만 의도하지 않은 예약을 만들 수 있음.

### Decision 7

### D-8. 예약 변경의 원자성 경계

## 4. GWT 명세

표기: 슬롯 = (일자, 시간, 테마) 조합. 날짜는 모두 "오늘 이후"를 전제로 하며 과거 케이스는 4.4에서만 다룬다.

### 4.1 테마 관리 (관리자)

```
시나리오: 테마를_생성한다
Given 관리자가 인증되어 있다
When 이름·설명·썸네일URL로 테마 생성을 요청한다
Then 테마가 생성되고 201과 생성된 테마가 반환된다

시나리오: 예약이_존재하는_테마_삭제를_거부한다
Given 테마 T를 참조하는 예약이 1건 존재한다
When 관리자가 T 삭제를 요청한다
Then 삭제가 거부되고 4xx와 에러 코드가 반환된다

시나리오: 예약이_없는_테마를_삭제한다
Given 테마 T를 참조하는 예약과 대기가 없다
When 관리자가 T 삭제를 요청한다
Then T가 삭제된다
```

### 4.2 시간대 관리 (관리자)

```
시나리오: 시간대를_생성한다
Given 관리자가 인증되어 있다
When 10:00으로 시간대 생성을 요청한다
Then 시간대가 생성된다

시나리오: 예약이_존재하는_시간대_삭제를_거부한다
Given 시간대 10:00을 참조하는 예약이 존재한다
When 관리자가 10:00 삭제를 요청한다
Then 삭제가 거부되고 4xx가 반환된다
```

### 4.3 예약 가능 시간 조회

```
시나리오: 예약된_시간이_제외된_가능_시간_목록을_반환한다
Given 등록된 시간대가 [10:00, 12:00, 14:00]이다
And 6월 15일 테마A의 10:00에 예약이 존재한다
When 6월 15일 테마A의 가능 시간을 조회한다
Then [12:00, 14:00]이 반환된다

시나리오: 다른_테마의_예약은_가능_시간에_영향을_주지_않는다
Given 6월 15일 테마B의 10:00에 예약이 존재한다
When 6월 15일 테마A의 가능 시간을 조회한다
Then 10:00이 가능 시간에 포함된다
```

### 4.4 예약 생성

```
시나리오: 가능한_슬롯에_예약을_생성한다
Given 슬롯 (6월 15일, 10:00, 테마A)에 예약이 없다
When 사용자가 본인 이름으로 해당 슬롯 예약을 신청한다
Then 예약이 생성되고 201이 반환된다

시나리오: 동일_날짜_시간이라도_테마가_다르면_예약된다
Given (6월 15일, 10:00, 테마A)에 예약이 존재한다
When (6월 15일, 10:00, 테마B)에 예약을 신청한다
Then 예약이 생성된다

시나리오: 중복_예약을_거부한다
Given 슬롯 S에 예약이 존재한다
When 다른 사용자가 S에 예약을 신청한다
Then 거부되고 4xx와 중복 에러 코드가 반환된다

시나리오: 지난_날짜의_예약을_거부한다
Given 오늘이 6월 10일이다
When 6월 9일 슬롯에 예약을 신청한다
Then 거부된다

시나리오: 오늘이지만_지난_시간의_예약을_거부한다

When (6월 10일, 10:00) 슬롯에 예약을 신청한다
Then 거부된다
```

### 4.5 내 예약 조회 / 취소 / 변경

```
시나리오: 본인_이름의_예약_목록을_조회한다
Given 갑의 예약 2건과 을의 예약 1건이 존재한다
When 갑이 본인 예약 목록을 조회한다
Then 갑의 예약 2건만 반환된다

시나리오: 본인_예약을_취소한다
Given 갑의 예약이 존재하고 해당 슬롯에 대기가 없다
When 갑이 예약을 취소한다
Then 예약이 삭제된다

시나리오: 지난_예약의_취소를_거부한다
Given 갑의 예약 일시가 현재보다 과거다
When 갑이 취소를 요청한다
Then 거부된다

시나리오: 비어있는_슬롯으로_예약을_변경한다
Given 갑의 예약이 슬롯 S1에 있고 슬롯 S2가 비어 있다
When 갑이 예약을 S2로 변경 요청한다
Then 예약 슬롯이 S2가 된다

시나리오: 점유된_슬롯으로의_변경을_거부한다
Given 슬롯 S2에 타인의 예약이 존재한다
When 갑이 본인 예약을 S2로 변경 요청한다
Then 거부되고 갑의 예약은 원래 슬롯을 유지한다

시나리오: 지난_예약의_변경을_거부한다
Given 갑의 예약 일시가 현재보다 과거다
When 갑이 변경을 요청한다
Then 거부된다
```

`[D-8 의존]` 변경으로 비워진 원래 슬롯의 대기 승격 여부·시점.

### 4.6 대기

```
시나리오: 예약된_슬롯에_대기를_신청하면_다음_순번이_부여된다
Given 슬롯 S에 예약 1건과 대기 1건(1번)이 존재한다
When 을이 S에 대기를 신청한다

When 갑이 S에 다시 대기를 신청한다

시나리오: 본인_대기를_취소한다
```

## 5. 상태 전이표

작성 범위 기준: 생애주기 상태를 갖고, 전이에 조건·부수효과가 있는 엔티티만. Theme·TimeSlot은 상태가 없으므로(존재/부재) 제외.

### Reservation

| 현재 상태 | 이벤트 | 가드 조건 | 다음 상태 | 부수효과 |
|---|---|---|---|---|
| (없음) | 예약 신청 | 슬롯 미점유 ∧ 미래 일시 ∧ 입력 유효 | 예약됨 | — |
| (없음) | 대기 승격 [D-1] | 슬롯 미점유 | 예약됨 | 잔여 대기 재정렬 [D-2] |
| 예약됨 | 취소 | 미래 일시 ∧ 본인 | (소멸) | 대기 1번 승격 [D-1], 재정렬 [D-2] |
| 예약됨 | 변경 | 대상 슬롯 미점유 ∧ 미래 일시 | 예약됨(슬롯 변경) | 원 슬롯 대기 처리 [D-8] |

### Waiting

| 현재 상태 | 이벤트 | 가드 조건 | 다음 상태 | 부수효과 |
|---|---|---|---|---|
| (없음) | 대기 신청 | 슬롯 점유됨 [D-7] ∧ 동일인 미대기 | 대기중(n) | 순번 = 현재 최대 + 1 |
| 대기중(n) | 본인 취소 | 본인 | (소멸) | 후순위 재정렬 [D-2] |
| 대기중(1) | 승격 [D-1] | 슬롯이 비워짐 | (소멸) | Reservation 생성, 후순위 재정렬 |

## 6. 동시성 시나리오

| ID | 시나리오 | 경합 자원 | 방어 전략 | 상태 |
|---|---|---|---|---|
| C1 | 동시 예약 신청 | reservation 슬롯 유일성 | UNIQUE(date, time_slot_id, theme_id) 최종 방어. 충돌 시 "대기 신청 안내"로 응답 (R-4) | 결정됨 |
| C2 | 승격 vs 신규 예약 | 비워진 슬롯 | 정책 자체가 미결 (D-4). 정책 확정 후 C1과 동일한 제약 기반으로 설계 | D-4 의존 |
| C3 | 대기 취소 vs 해당 대기의 승격 | waiting 행 | 승격 트랜잭션과 취소 트랜잭션의 직렬화. 테이블 통합(D-9) 채택 시 단일 행 UPDATE 경합으로 축소 | D-9 의존 |
| C4 | 동시 대기 신청의 순번 경합 | MAX(rank) 읽기 | UNIQUE(slot, rank) 최종 방어 + 충돌 시 재시도 (R-5) | 결정됨 |
| C5 | 동시 취소 2건의 재정렬 경합 | 동일 슬롯 waiting 순번 집합 | 슬롯 단위 직렬화 필요. D-2 의미론 확정 후 설계 | D-2 의존 |
| C6 | 관리자 시간대/테마 삭제 vs 예약 생성 | 참조 무결성 | FK 제약이 최종 방어 (R-1). SELECT-then-DELETE 검증의 TOCTOU를 FK가 흡수 | 결정됨 |
| C7 | 예약 변경 vs 대상 슬롯의 신규 예약 | 대상 슬롯 유일성 | C1과 동일한 UNIQUE가 방어. 단 변경의 원자성 경계(D-8)에 따라 "원 슬롯 소실" 위험이 달라짐 | D-8 의존 |

## 7. 결정 기록 (무효화 트리거 포함)

### R-1. 물리 FK 채택
- **결정**: reservation/waiting → theme, time_slot 참조에 물리 FK 사용.
- **유효 조건**: 단일 DB 인스턴스, 무중단 스키마 변경 도구(gh-ost 등) 불요, 쓰기 락 경합 미관측.
- **무효화 트리거**: 샤딩·서비스 분리 착수, FK 검증 기인 락 대기 알림 발생, 파티셔닝 필요 — 발생 시 논리 FK + orphan 탐지 배치로 이행 재검토.
- **부수 이득**: C6의 TOCTOU를 DB가 흡수.

### R-2. 도메인 접두어 생략
- **결정**: RoomescapeTheme이 아닌 Theme. 관례·해석 용이성·표기 비용.
- **무효화 트리거**: 동일 바운디드 컨텍스트에 이름이 충돌하는 제2의 Theme 개념 등장.

### R-3. 공유 참조 데이터의 테이블 분리
- **결정**: 테마·시간을 각 사용처 컬럼 중복이 아닌 독립 테이블 + 참조로 관리.
- **근거**: 갱신 이상 제거, 단일 진실 공급원.

### R-4. 동시 예약 방어 = UNIQUE 최종 방어 + 안내 응답
- **결정**: 애플리케이션 사전 검증(친절한 에러)과 무관하게 UNIQUE(date, time_slot_id, theme_id)가 최종 방어. 제약 위반 시 사용자에게 대기 신청을 안내.
- **무효화 트리거**: D-9 테이블 통합 채택 시 제약 위치 재설계(NULL 보조 컬럼 우회 검증 필수).

### R-5. 순번 경합 방어 = UNIQUE(slot, rank) + 재시도
- **결정**: MAX+1 경합은 UNIQUE로 차단하고 충돌 시 재시도.
- **무효화 트리거**: D-2에서 결번 유지(b)를 채택하면 재정렬과의 상호작용 재검토.

### R-6. 대기 → 예약 전환 방식
- **결정**: 예약 취소 시 자동으로 가장 빠른 대기를 찾아 예약으로 승격. 
- **유효 조건**: 예약과 대기가 같은 자원(날짜,시간,테마)를 정확히 공유하고 대기한 사용자에게 자동 전환을 고지한 경우. 
- **무효화 트리거**: 예약의 취소와 대기의 승격이 동시에 보장되지 않아도 된다는 요구사항이 명시될 경우, 공유 자원이 불일치하게 변경된 경우

### 전제 (Premises)
- **P-1**: 모든 테마의 시작 시간·소요 시간 동일. — 무효화 시 TimeSlot이 Theme 종속으로 재설계되어 §3 전체에 파급.
