## 트랜잭션 경계와 동시성 — 예약 취소와 대기 자동 승격

`상황` - 취소와 승격은 하나의 트랜잭션인가?  
`결정` - **취소와 승격을 하나의 트랜잭션으로 묶었다.**  
`이유` - **경계의 기준은 DB 구조가 아니라 비즈니스 정합성이다.**

취소 시 예약만 삭제되면 "예약 없는 대기"라는 비정합이 사용자에게 노출된다. 두 변경은 하나의 비즈니스 결과다.  
반대로 테이블이 여러 개 바뀐다는 기술적 이유만으로는 묶지 않는다.

`상황` - 공통 개념을 추출해 관리해야 하는가?  
`결정` - **공통 개념(날짜·시간·테마)을 불변의 session으로 추출해 예약 1 : 세션 1 : 대기 N 구조로 정규화했다.**  
`이유` - **DB 구조를 비즈니스 정합성을 보장의 도구로 활용하기 위해**

중복된 데이터를 최소화하고, 도메인 로직에서 관리하기 어려운 경합 상황을 DB에서 제약조건으로 해결한다.

**동시 취소가 일으키는 중복 승격은 UNIQUE(session_id)가 막는다 — 정규화가 만든 방어선.**  
실험에서 두 트랜잭션이 같은 1순위 대기를 읽고 승격을 시도하는 경합을 발견했다.  
H2(READ COMMITTED)의 MVCC 비잠금 읽기가 같은 대기를 읽도록 허용한다.  
영향 행 수를 검사하지 않는 현재 구현에서 UNIQUE 제약이 유일한 방어선임을 코드로 확인했다.  
제약은 "한 슬롯에 예약은 하나"라는 불변식을 DB가 검사하는 형태로 번역해, 경합의 패자를 롤백시킨다.

> **트랜잭션 경계 한 줄** — 비즈니스 비정합 상태를 사용자에게 노출할 작업들은 하나의 트랜잭션으로 묶는다.

## 결정 2 · 에러 응답 — 잘못된 요청에 합당한 응답

**에러 응답을 code + message로 표준화하고, 예측 가능한 위반은 4xx로 변환했다.** 존재하지 않는 예약 조회와 중복 예약이 500과 무의미한 본문으로 노출되고, 지난 날짜 예약은 201로 성공하고
있었다. 원천 데이터의 검증 책임이 서버에 있으므로, 실패의 원인과 대응을 응답에 일원적으로 담는다.

**"다음 행동" 정보를 위한 추가 조회는 금지했다.** 응답 정보는 요청-응답 흐름 내부로 한정한다. 가용 시간 안내는 기존 조회 기능의 책임이다. 올바른 요청에 올바른 응답이면 동작하는 API, 잘못된 요청에도
합당한 응답이면 바람직한 API다.

## 결정 3 · 테스트 — 보호할 책임의 분리

**테스트가 보호하는 것은 프레임워크가 아니라 우리가 작성한 비즈니스 규칙과 외부 동작 계약이다.** 레벨 1식 순수 단위 테스트가 서비스 계층에서 막혔고, 원인을 "도메인 서비스가 아닌 애플리케이션 서비스"로
진단했다. 책임의 종류가 테스트 단위를 결정한다.

**검증 위치와 도구는 우선순위로 정한다.** 외부 의존 없이 가능하면 단위, 결합 자체가 대상이면 통합·슬라이스, 사용자 흐름 보호면 최소한의 E2E. 더블은 호출 행위면 Mock, 쿼리·매핑이면 인메모리 DB,
벤더 특화면 실제 DB. 단순 위임 계층과 프레임워크 보장 영역은 테스트하지 않는다. 테스트는 꺼려지는 의무에서 프로덕션 코드와 같은 유지보수 대상으로 바뀌었다.

## 학습 과정

**규칙은 정답이 아니라 기준을 갈고닦는 보조 도구다.** 사전 학습 → 토론 → 구현 → 피드백 사이클을 나선형으로 반복했고, 학습법 자체도 Feedback Loop — "고민보다 실패를" — 를 축으로 v1에서
v3까지 튜닝했다.
