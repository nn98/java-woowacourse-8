### `CQS`

> TOP 인 CQS 의 근본적 목표를 먼저 확인하고,  
> Bottom-UP 방식으로 타고올라가며 연결된 개념을 이해하기

`P` - TOP, 근본적 목적 추론하기

상태를 수정하는 `명령` 과 상태를 조회하는 `쿼리` 를 분리해야 한다.

- 두개가 혼재되는 순간 코드의 의도와 구현에 대한 불명확성이 발생하고,
- 이는 클린하지 못한 코드이자 코드의 유지보수성/가독성을 모두 저해하기 때문.

> 방향성과 전제는 옳다

`i` - bottom, 도입 목적 상세히, 부분적으로

- 창시자 버트란드 마이어
- 객체의 메서드가 수행하는 행위를 두개의 분류로 구분
    - 상태를 변경한다 - 명령 `Command`
    - 상태를 반환한다 - 조회 `Query`

`i` - up, 구분된 각각에 대한 활용과 원칙

- 명령 `Command`
    - 객체의, 나아가 시스템의 상태를 변경하지만 값을 반환하지 않는다.
    - 왜?
    - SRP 관점에서 변경과 조회가 같은 책임인가?
    - 그럼 Collection.remove 에선 왜 값 반환함? 머임? -> `마틴 파울러의 '실용적 예외' 허용`
- 조회 `Query`
    - 객체의, 나아가 시스템의 상태를 조회하지만 값을 변경하지 않는다.
    - 왜?
    - SRP 관점에서 변경과 조회가 같은 책임인가?
    - 조회할 때마다 상태가 변한다면 객체의 무결성이 보장되는가?
- 그 둘을 분리함으로써 얻는 이점
    - 조회는 상태의 정합성을 보장할 것이라는 믿음과 신뢰를 주고,
    - 이를 기반으로 개발자의 코드에 대한 리뷰와 코드에 대한 해석을 원활하도록 한다.

`i` - up, 원칙에 대한 반론 : 엄격한 CQS 준수의 비효율성

- Case 1: Stack - `동시성` 버그
    - 스택에서 pop 하면 상태의 변경과 조회를 동시에 한다
    - 왜?
    - 멀티스레드 환경에서 값이 하나만 있는 스택에 동시에 pop 한다면?
    - 위와 같은 상황에서 pop 이 조회 따로 명령 따로 진행한 경우와  
      변경과 조회를 동시에 진행한 경우, 어떤 경우가 객체의 무결성을 보장하는가?
- Case 2: DB AutoIncrement
    - DB에 값을 Insert 하는 것은 명령이다
    - Insert 를 호출한 곳에선 삽입될 데이터의 ID 를 모를 수 있다(DB 에서 AutoIncrement)
    - 이 때 반환값이 없다면 insert 를 호출한 곳에선 그 결과를 어떻게 확인하는가?
    - 호출한 정보를 기반으로 insert 한 정보를 다시 조회해서 확인해야 하는가?
    - 위와 같은 방법과, insert 의 결과로 ID 를 반환하는 방법 중  
      어떤 방법이 더 효율적이고 안정적인가?

`i` - up, 원칙에 대한 개선책 : `마틴 파울러의 '실용적 예외' 허용`

- 주창자 마틴 파울러의 `실용적 예외`
    - 명령이라도 원자성 보장이 필수적일 경우, 값을 반환할 수 있다
    - 명령의 결과를 알아야 할 경우, 결과값을 반환할 수 있다
- 마틴 파울러의 설계 철학
    - 학술적이고 교조적인 완벽함을 배척
    - 설계 원칙은 `유지보수성` 과 `실용성` 을 위한 도구일 뿐,
    - 그 자체가 목적이 되어선 안된다
    - 본말전도를 항상 경계하고 근본적 목적을 파악하라

`F` 먼저 실천했으니 개념적 필요성을 이해하기

`SimpleJdbcInsert.executeAndReturnKey`  
깔끔하게 Insert 결과로 생성된 ID 반환, 그 값으로 생성된 결과에 접근 가능

`S` 총체적인 목적을 먼저 파악하고 Bottom-Up 탐구했으니 구조적 학습

`R` 내 경험에 기반한 명확한 반례와 생각에 대한 AI의 검증 활용

### 스스로 답을 찾아낼 비판적 질문 (학습 내용의 한계 시험)

이번 학습을 통해 CQS 원칙과 데이터베이스 INSERT 시 발생하는 실용적 예외(`SimpleJdbcInsert.executeAndReturnKey`)의 상관관계를 완벽하게 이해하셨습니다.

이제 이 개념을 객체지향 영속성 프레임워크인 JPA(Hibernate)로 확장해 보겠습니다.

JPA의 핵심 저장 메서드인 `EntityManager.persist(entity)`는 놀랍게도 CQS 원칙을 엄격하게 준수하여 **반환값이 `void`**입니다. JDBC 환경에서 극찬하셨던 그 유용한 ID 반환(
`return id;`)을 의도적으로 없애버린 것입니다.

그렇다면 JPA 환경에서는 방금 저장하여 DB에서 `AUTO_INCREMENT`로 발급받은 그 새로운 식별자(ID)를 서비스 계층에서 어떻게 알아내어 후속 비즈니스 로직(예: 저장된 ID로 다른 객체와 연관 관계
맺기)에 활용할 수 있습니까? 반환값이 없는 상황에서 ID를 획득하기 위해 JPA가 객체(Entity)의 상태 관리를 어떠한 방식으로 통제하는지 그 내부 메커니즘을 분석해 보십시오.
