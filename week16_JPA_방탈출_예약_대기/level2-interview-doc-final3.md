# 레벨 2 회고 — 기술적 의사결정과 설계 경험

> ## 레벨 2 의사결정의 핵심 기준
>
> - **이론과 실제의 타협**.
>   요구사항 충족을 최우선에 두고, 책임을 분배하고 역할을 응집해 유지보수성을 높인다.
>   이론과 원칙을 언제 따르고 언제 버릴지는 상황과 조건, 비즈니스 요구사항에 따라 결정된다.
>   객체 지향적 구조 위에 절차 지향적 흐름을 설계한다.
>
> 같은 기준을 먼저 세웠기에, 리뷰어의 지적도 수정이 아닌 기준을 다듬는 재료로 흡수했다.
> 동일 기준의 적용 사례 세 가지.

## 사례 1 · 이론을 버린 타협 — 참조·조립·순번의 실용적 예외

`상황` 도메인을 DDD 정의대로 식별자 참조로 끊을 것인가?
`결정` **식별자 참조를 버리고 객체 기반 참조를 택했다.**
`이유` **이론을 위배하더라도 객체가 스스로 책임을 수행하는 것이 올바른 객체 지향이다.**

식별자로 끊으면 `Reservation` 이 행위할 때마다 `Time`·`Theme` 를 다시 조회해 조립해야 한다.
모든 도메인이 애그리거트 루트 조건을 충족하는 현 구조에서, 이론적 정의보다 객체의 자율성을 우선했다.
파울러의 `실용적 예외` 다.

객체를 불변으로 둘 수 있었으나 `DDD` 의 엔티티 정의에 끌려 가변으로 설계했다.
"JDBC 환경에서 가변 엔티티는 변경 시점 추적이 어렵다"는 리뷰를 받고 불변으로 교정했다.
자율적 결정은 유지하되, 이론에 끌려간 한 곳은 피드백으로 바로잡았다.

도메인 로직의 실행 위치도 같은 기준으로 반복됐다.
예약 가능 시간 계산, 도메인 조립, 대기 순번 산출을 자바 메모리가 아닌 조인·집계 쿼리로 `DB` 에 맡겼다.
`리소스`·`도메인`·`스키마` 가 1:1:1로 대응하지 않는 한, 조립에는 도메인 로직이 녹아든다.
"영속성 매핑일 뿐"이라 방어하지 않는다.
`N+1` 을 피하기 위해 도메인 로직이 쿼리로 새어든 `실용적 예외` 임을 인정한다.

## 사례 2 · 끝까지 시도하고 같은 기준으로 철회 — 동시성과 계층적 락

`상황` 동시 취소의 경합을 어떻게 막을 것인가?
`결정` **계층적 락을 시도했으나 철회하고, UNIQUE 제약으로 수습했다.**
`이유` **계층적 락의 전제인 애그리거트 루트 구조가 현 도메인에 적합하지 않았다.**

데드락 없는 동시성 제어를 위해 `Slot` 을 애그리거트 루트로 올리려 했다.
실제로 `Slot` 을 `Session` 으로 분리해 코드까지 바꿨다.

리뷰어가 짚었다 — "별도 테이블과 생명주기 공유라는 두 방향을 동시에 채택해 복잡하다".
코치가 물었다 — "지금 구조라면 오히려 예약이 애그리거트 루트인데 왜 슬롯을 루트로 보는가".

하위 엔티티를 거느릴 구조가 아니었다.
스키마와 도메인 객체의 분리는 남기고, 애그리거트 루트라는 개념의 도입만 철회했다.
스스로 깨달은 것이 아니다.
타협의 기준을 갖고 있었기에 피드백을 수용해 본질로 돌아갔다.

수습은 구조를 바꾸지 않고 가능했다.
동시 취소 시 두 트랜잭션이 같은 1순위 대기를 읽고 승격을 시도한다.
이 경합을 정규화로 선언 가능해진 `UNIQUE(session_id)` 가 막는다.

원자성은 지켜졌다. 원인은 격리다.
`H2`(READ COMMITTED)의 `MVCC` 비잠금 읽기가 같은 대기를 읽도록 허용한다.
영향 행 수를 검사하지 않는 현재 구현에서, 이 제약이 유일한 방어선임을 코드로 확인했다.
제약은 "한 슬롯에 예약은 하나"라는 불변식을 `DB` 가 검사하는 형태로 번역해, 경합의 패자를 롤백시킨다.

취소와 승격을 하나의 트랜잭션으로 묶었기에 이 수습이 의미를 가진다.
취소만 성공하면 "예약 없는 대기"가 노출된다 — 두 변경은 하나의 비즈니스 결과다.
테이블이 여러 개 바뀐다는 기술적 이유만으로는 묶지 않는다.

> **트랜잭션 경계 한 줄** — 부분 실패가 사용자에게 비정합 상태를 노출할 수 있는 작업들은 하나의 트랜잭션으로 묶는다.

## 사례 3 · 실패는 계층이 아니라 흐름에 속한다 — 예외의 위치

`상황` 예외를 계층별로 나눠 관리할 것인가?
`결정` **계층이 아니라 실패의 의미(흐름)를 기준으로 예외를 관리한다.**
`이유` **DTO는 계층 관점의 데이터 전달, 예외는 흐름 관점의 실패 표현이다.**

처음엔 `DTO` 처럼 예외도 발생한 계층에 종속된다고 보았다.
"계층별로 나누는 것은 추천하지 않는다, 왜 그렇게 생각했는지 먼저 설명해달라"는 역질문을 받았다.
스스로 근거를 따져보며 관점이 바뀌었다.

"예약을 찾을 수 없다"는 실패는 서비스에서 터지든 레포지토리에서 터지든 같은 의미다.
계층이 수평이라면 흐름은 수직이다.
실패는 흐름에 종속된 개념이다.

같은 자각이 테스트에서도 나왔다.
`Fake` 로는 통과하던 조회가 실제 `JDBC` 구현에서 깨졌다.
인터페이스라는 `계약` 을 `Fake` 와 실제 구현이 동일하게 지키는지 검사하지 않았다.
무엇이 어느 책임에 속하는지를 기준으로, 예외의 위치도 테스트의 대상도 결정했다.

## 학습 과정

타협의 기준은 한 번에 서지 않았다.

- `CQS` 의 명령/조회 분리에서 출발해, Stack pop과 AUTO_INCREMENT 반례로 `실용적 예외` 의 필요를 도출했다.
- `N+1` 을 직감으로 감지한 뒤, 조인과 조립의 책임 분배로 정리했다.
- `DDD` 의 목적을 "소프트웨어 중심부의 복잡성 통제"로 다시 세웠다.
- 테스트를 꺼려지는 의무가 아닌, 프로덕션 코드와 같은 유지보수 대상으로 인식했다.

사전 학습 → 토론 → 구현 → 피드백을 나선형으로 반복했다.
규칙은 정답이 아니라 기준을 갈고닦는 보조 도구로 다뤘다.
