## 사전 학습에서 생각한 부분

어휘나 표현력이 부족하다보니 간결명료하게 생각을 표현하기보단  
일단 기이이일게 작성하고 나중에 복기하면서 축약하곤 합니다..

간결하게만 읽어주시면서 핵심적인 부분에서 혹 잘못된 방향성이나  
수정해야 할 생각이 있다면 날카롭게 지적해주시면 너무나 감사드리겠습니다!

> ### 테스트를 작성하기 어려운 코드가 있는가?
>
> ```java
> @DisplayName("정상적 숫자/문자로 카드 생성")
> @ParameterizedTest
> @ValueSource(strings = {"2", "4", "7", "9", "Q"})
> void 정상적_숫자_문자로_카드_생성(String denomination) {
>      assertThatNoException()
>          .isThrownBy(() -> Card.findByDenomination(denomination));
> }
> ```
>
> - `사실`
    >   - 테스트 자체는 위와 같이 NoException으로 정상 동작 검증
> - `문제`
    >   - 생성 과정에서 예외 발생 유무는 검증이 가능.
>   - 하지만 예외 없이 생성된 각 개체가 내가 의도한 대로 생성되었는지,  
      >     그 내부에 대한 유효성 검증은 전무.
> - `예시`
    >   - 생성된 카드들로 점수 계산 시 예상한 결과가 나오지만 (2, 4, 7, 9, Q) → 32,  
          >     이 각각의 카드가 정말 2, 4, 7, 9, Q(10)이란 값을 가지고 있는 게 맞을까?
>
> ### 어렵다면, 무엇 때문에 어려운가?
> - `사실`
    >   - 현재 예외 발생의 유무로만 검증을 진행
> - `문제`
    >   - 정상 동작 여부 뿐만 아니라  
          >     동작의 결과로 생성된 객체의 유효성을 검증해야 함
>
> ### 테스트를 쉽게 하려면 코드를 어떻게 바꾸고 싶은가?
> - `사실`
    >   - 동작(행위) 자체의 유효성 검증은 당연하며 그 결과에 대한 유효성 검증도 필요
> - `문제`
    >   - 결과를 확인하기 위해 getter/setter 구현 시 결합도 상승,  
          >     단순 테스트용 equals 구현 시 도메인 로직과의 무관성 우려
> - `해결`
    >   - 단순히 값을 저장만 하는 클래스가 아닌 VO(Value Object)로 기능을 확대.
> - `근거`
    >   - 해당 값의 유효성을 확인하고 보유하는 로직은   
          >     도메인에 '무관하지 않은' 필수 로직이며,  
          >     이를 통해 TDA(Tell, Don't Ask) 원칙 준수 및  
          >     객체 간 책임 분리를 통한 명확한 검증 구현 가능

페어 활동에서 가장 고민하고 궁금했던 내용을 정리하면

### 1. 에이스(1점으로도, 11점으로도 사용 가능) 처리 방식

- enum 이나 상태로 관리해서 계산 중 상태를 반영?
- enum Rank 에서 Ace 면, 상태를 isUsed 를 확인해서
  - 11로 사용할 에이스인지, 1로 변경한 에이스인지 확인, 반영
- 규칙을 행위의 주체가 인지하고, 행위 과정에서 판별?
  - 매 계산을 진행할 때 마다 에이스를 순회하며 -10 처리

### 2. 어떤 플레이어의 턴인지 판별

- 플레이어의 순서를 정해서(Map<Integer, Player>) 순회하며 게임 진행?
- 플레이어는 일급 컬렉션에서 관리, 일급 컬렉션이 순서를 판정?

현재 구현한 방식은
- 판정 방식은 컬렉션만의 순서로 알아서 판별, `Players - findDrawablePlayer()`
- 판별 결과도 알리지 않으려 했으나  
  누구 차례인지 출력해야해서 닉네임만 반환, 원본 접근 제어
- 더 뽑을지 말지 입력받고 `private boolean isDraw(String nickname)`
- 더이상 입력을 안받을거면
  - 해당 차례의 플레이어가 `Players - findDrawablePlayer()`
- 카드를 받지 않음을 업데이트 `dontWandDraw()`
- 뽑을거면
  - 해당 차례의 플레이어가 `Players - findDrawablePlayer()`
  - 카드를 뽑으라고 덱을 전달.
- 컬렉션은 항상 동일한 판별방식으로 해당 차례의 플레이어를 찾도록 보장.

이라는 좀 불안?한 방식인데 혹여나 수정사항이나 보완사항이 있을지 궁금합니다!

### 3. 일급 컬렉션의 책임? 역할?

위 질문과 연계되는 질문이라 생각하는데

- 카드의 점수 합계는 카드 객체의 컬렉션인 PlayingCards 가 담당하는가?
- 그 PlayingCards 를 필드로 가진 Player 가 담당하는가?


- 카드의 점수 합계가 버스트인지는 플레이어가 담당하는가?
- 그 플레이어들을 관리하는 컬렉션인 Players 가 담당하는가?


위와 같이 정리할 수 있을 것 같습니다.  
컬렉션이 단순 접근 제어만 하는 것은 과소하고  
관련된 모든 로직과 행위를 알고 수행하는 것은 과도하다 생각해
