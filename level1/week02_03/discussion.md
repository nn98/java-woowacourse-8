- 글렌
    - **테스트 단위 기준**
        - `최소 단위 테스트` (이전)
          `메서드. 가장 작은 기능, 하지만 명확히 사용되는 기능
          사용되지 않거나 해당 단위에서 테스트가 어렵다면 
          해당 책임을 가지는 단위로 이관`
        - 핵심은 최소 단위, 점진적 확대
            - 가장 작은 기능, 메서드 단위로 테스트
            - 단, 서비스에서 명확히 사용되는 기능에 한정
            - 가장 낮은 단위, 기본적 기능을 테스트
            - 상위 기능 == 하위 기능의 조합 이라면,
              더 작은 단위에서 검증해야 한다
    - 테스트 포함/제외 기준
        - `테스트 제외` (이전)
          `IO 테스트, 실제 서비스 동작에 사용되지 않는 기능 
          IO 테스트가 테스트할 내용은 모두 단위 / 기능 테스트 단에서 
          처리되었어야 하며, 실제 동작에 사용되지 않는 기능은 
          명확한 근거 없이는 테스트도, 존재 이유도 없다`
        - 테스트 포함
            - 최소 단위의 **`비즈니스 로직`**
              **-> `도메인 객체의 핵심 비즈니스 로직과 상태 변경.`**
                - 베이스 기능이 명확히 동작해야 그 상위 기능의 정상적 동작을 보장
                - 가장 직관적으로, 직접적으로 테스트 가능
                - 테스트와 설계에 대해 공부한다는 핵심 목표를 고려할 때,
                  최소 단위 테스트가 가장 바람직

          **`비즈니스 로직`이란 불명확한 표현보단**
          대상에 대한 명확한 표현과 구체화로 특정성과 방향성을 확립

        - 테스트 제외 (이후)
            - IO 로직
                - IO 단의 로직은 도메인의 핵심 비즈니스 로직과 무관한 UI 영역.
                  UI 로직의 결과 데이터에 대한 예외나 후속 동작은 테스트가 필요하나
                  UI 로직 자체를 테스트하는 것은 불필요
                - 상술한 후속 동작, 입출력 로직 이하의 기능적 / 구조적 동작은
                  `테스트 포함` 항목의 최소 단위 테스트에서 검증 / 처리
            - 실제 비즈니스 로직에 적용되지 않는 기능
                - 실제 서비스에서 사용되지 않는 기능은
                  그 존재 목적을 명확히 판별해
                  필요하다면 그 기능을 서비스에서 활용하도록 하고,
                  불필요하다면 기능 자체를 제거해야
            - YAGNI(You Aren't Gonna Need It) 원칙을 준수하는 방식으로
    - 테스트 어려움 대응
        - `보예` 의견 반영,
        - 기능 분리와 책임 이관을 통한 원인 제거
        - 테스트의 어려움은 해당 기능의
          중첩/복합에 기원. 기능의 분리와 책임 이관을 통해
          그 근본적 원인을 제거함으로써 대응해야
    - 리팩터링 우선순위
      컨벤션/원칙 > 기능 > 로직 > 코드?
      지금 가장 핵심으로 생각하고 고민하는 내용은 원칙과 기준이라 생각.

- 캐모
  도메인 단위 테스트.
  도메인 단위를 벗어난 내용은 테스트 X
  큰 틀에서 테스트할 내용을 작은 부분에서 하는 경우도.
  책임 분리 > 로직 개선 > 가독성

- 우디
  도메인 내부 메소드 단위
  입출력 등 단위를 넘어선 부분은 X
  랜덤값 같은 부분은 인자값 받아서 테스트
  로직 단순화 > 역할 축소 > 역할 분리

> 컨벤션 > 역할 분리 > 로직

1. 테스트 단위 기준
   가장 작은 단위, 메소드

2. 테스트 제외 기준
   IO 테스트, 중복되거나 하위 부분

3. 테스트 어려움 대응
   기능 분리와 역할 이관

4. 리팩터링 우선 순위
   컨벤션/규칙 > 역할 분리 > 세부 로직
