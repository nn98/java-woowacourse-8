`장기`

|  순번  | 기준                              | IF(만약 ~ 라면)                                                  | THEN(~한다)                                                |
|:----:|---------------------------------|--------------------------------------------------------------|----------------------------------------------------------|
| `1`  | 상태 위치 관리                        | 만약 기물의 위치가 이동, 잡힘, 배치 변경처럼 <br/>게임판 전체 상태 변화와 함께 바뀐다면        | 위치 정보는 보드가 관리하고, <br/>기물은 자신의 위치를 모른다                    |
| `2`  | 불변 객체 기준                        | 만약 기물이 가지는 정보가 <br/>종류, 소속 팀, 이동 규칙처럼 <br/>게임 도중 바뀌지 않는 값이라면 | 기물 객체는 불변으로 만든다                                          |
| `3`  | 캡슐화 기준                          | 만약 외부 직접 접근이 필수적이지 않다면 <br/>또한 그 접근이 객체의 정합성을 저하한다면          | 객체의 상태와 행위는 캡슐화하고, <br/>외부에선 그저 요청하고 호출한다                |
| `4`  | null 사용 기준                      | 만약 빈 칸도 보드 위의 의미 있는 상태라면                                     | 빈 칸은 null 대신 객체 또는 <br/>존재 여부를 확인하는 방식으로 표현한다            |
| `5`  | 조건문 대체 기준                       | 만약 조건문이 객체의 종류에 따라 <br/>동작을 나누기 위해 사용된다면                     | 조건문 대신 다형성을 우선 고려하고, <br/>각 객체가 자신의 규칙을 직접 구현한다          |
| `6`  | 역할/인터페이스 <br/>설계 기준             | 만약 여러 객체가 같은 행위를 하지만 <br/>구현 방식이 서로 다르다면                     | 인터페이스로 공통 행위를 정의하고 <br/>구현체로 세부 내용을 구현한다                 |
| `7`  | 새 타입 추가 시 <br/>변경 범위 <br/>제한 기준 | 만약 새 기물이 추가될 때 <br/>기존 이동 로직을 수정해야 한다면                       | 기물별 책임 분리가 미흡한 것이니 <br/>이동 로직을 명확히 분리한다                  |
| `8`  | 도메인/저장소 분리 기준                   | 만약 도메인 객체가 DB 연결 정보나 SQL 쿼리를 직접 알고 있다면                       | 인터페이스를 도입하여 도메인 로직과 데이터 액세스 로직을 엄격히 분리한다.                |
| `9`  | 저장 경계 규칙                        | 만약 저장/조회 로직이 장기판의 이동 로직이나 승패 판정 로직과 섞여 있다면                   | 서비스(Service) 계층을 도입하여 도메인 객체에 행위를 시킨 후, 그 결과를 저장소에 전달한다. |
| `10` | 저장 시점 규칙                        | 만약 특정 행위(이동, 잡기 등)가 도중 실패하거나 예외가 발생한다면                       | 데이터를 저장하지 않고 이전 상태를 유지하는 트랜잭션 롤백 원칙을 적용한다.               |

## 기능 요구 사항

### 2.1단계 - 기물의 확장
- 사이클1에서 제외했던 궁성(宮城) 영역의 규칙을 포함하여 기물의 이동을 구현한다.
- 장기는 왕이 잡히는 경우 게임에서 진다. 왕이 잡혔을 때 게임을 종료해야 한다.
- **현재 남아 있는 기물에 대한 점수를 구할 수 있어야 한다.**
- 점수 계산 규칙은 직접 요구사항을 분석하여 정의한다.

- DB를 적용할 때 도메인 객체의 변경을 최소화해야 한다.

## 레코드의 활용법과 좋은 getter

[레코드](https://openjdk.org/jeps/395)를 주제로 크루들과 토론 후 내린 나름의 결론은
- 레코드는 VO(값 객체)로서,
- 그 V(값) 에 대한 행위나 책임을 보유하고,
- 그 V(값) 에 대한 무결성과 유효성을 보장함으로써
- 높은 응집도의 도메인 객체 역할을 수행할 수 있다

였습니다.

이를 통해
- 불변 클래스를 record 로 변환

- 방어적 복사를 적용
- view 로 전달 시 DTO 로 포장
