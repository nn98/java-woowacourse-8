`N+1 쿼리 문제`

- 도메인 객체의 필드가 다른 도메인 객체일 때, 부모 객체 목록을 조회(`findAll`)한 뒤  
  각 객체 자식 데이터를 매핑하기 위해 N번의 추가 쿼리가 발생하는 현상
- 단 1번의 API 호출이 100번, 1000번의 DB 네트워크 통신(Round-Trip)을 유발하여  
  트래픽 증가 시 즉각적인 데이터베이스 마비 및 장애로 직결됨

`i` - up, 해결책: RDBMS의 본질 `JOIN`

- 메모리 단에서 DAO를 각각 호출하여 루프를 도는 것보다,  
  `INNER JOIN`을 통해 단 1번의 네트워크 통신으로 조립에 필요한 모든 컬럼 데이터를
  가져오는 것이 압도적으로 자원 효율적이다
- 미션에서부터
- > 여러 테이블에서 일치하는 데이터를 조회할 때 inner join 키워드를 활용한다. 스스로 작성할 수 있도록 연습한다.

`i` - append(추가), 한계: 분산 환경(MSA)

- 물리적으로 예약 DB와 시간 DB가 분리된 서버 아키텍처라면 SQL 단위의 JOIN은 물리적으로 불가능
- 이 경우 불가피하게 API Gateway나 애플리케이션 계층에서 2번의 네트워크 통신을 거쳐 메모리에서 조립해야 하며,
- 이로 인한 지연(Latency)을 막기 위해 비동기 병렬 호출이나 Redis 캐싱 등의 방어 로직이 강제된다.

### 1. 전략 평가 및 구조적 분석

**강점 (효과적이었던 부분):**

1. **직감을 객관적인 근거로 치환:** `RowMapper` 내부의 DAO 연쇄 호출을 보며 단순히 "코드가 지저분하다"고 넘기지 않고, "이게 SRP(단일 책임 원칙)를 준수하는가?", "조회 빈도가
   기하급수적으로 늘어나는 것이 아닌가?"라는 구체적인 아키텍처 관점의 질문으로 승화시켰습니다. 이러한 감각이 바로 성능 병목을 사전에 차단하는 시니어 개발자의 가장 중요한 자질입니다.
2. **명확한 트레이드오프(Trade-off) 인지와 책임 분리:** 메모리 단의 루프 조립과 DB의 `JOIN` 간의 본질적인 성능 차이(네트워크 I/O 비용 vs DB 엔진의 최적화)를 정확히 인지했습니다.
   나아가, 데이터의 '결합(JOIN)'은 RDBMS에, 그렇게 결합된 결과 셋을 도메인 구조로 '캐스팅(조립)'하는 것은 애플리케이션 계층에 각각 적절하게 책임을 분배한 점이 탁월합니다.
3. **미래 아키텍처(MSA)를 대비한 시야 확장:** 현재의 Monolithic 환경(단일 DB)에서 도출한 정답(`JOIN`)이, 물리적으로 DB가 찢어지는 MSA 환경에서는 오답이 될 수 있음을 인지한 것은
   극찬할 만한 성과입니다. 이는 기술에 대한 맹목적인 신뢰를 경계하고, 시스템의 규모와 환경에 따라 정답이 변할 수 있다는 유연한 아키텍트의 시각을 갖추기 시작했다는 증거입니다.

**보완점 (더 잘게 쪼개야 한다는 스스로의 평가에 대하여):**

### 스스로 답을 찾아낼 비판적 질문 (학습 내용의 한계 시험)

이번 학습을 통해 관계형 데이터베이스(RDBMS) 환경에서는 `JOIN`을 통한 일괄 조회가 애플리케이션 레벨의 조립(N+1 방식)보다 성능적으로 압도적인 우위에 있음을 증명하셨습니다.

하지만 객체지향의 세계에서는 도메인이 항상 `1:1` 또는 `N:1`의 단순한 관계(예: 예약과 시간)만을 가지지 않습니다. 방탈출 시스템에 **'하나의 예약(Reservation)이 여러 명의 참가자(
Participant) 목록을 가진다'**는 `1:N` 컬렉션 관계가 새롭게 도메인에 추가되었다고 가정해 보겠습니다.

이때 이전 학습의 교훈을 살려 1번의 쿼리로 모든 데이터를 가져오기 위해, `reservation` 테이블과 `participant` 테이블을 `INNER JOIN` 하는 단일 쿼리를 작성했습니다.

데이터베이스는 1개의 예약 데이터에 5명의 참가자가 존재할 경우, 이 `JOIN` 쿼리의 `ResultSet`으로 예약 정보(부모 데이터)가 중복된 채 총 몇 줄(Row)의 데이터를 반환하게 됩니까?
이처럼 `1:N` 조인 결과로 인해 발생하는 **'부모 데이터의 중복(Cartesian Product 현상)'**이 네트워크 전송량(대역폭) 측면에서 유발할 수 있는 새로운 성능 오버헤드는 무엇이며, 이를 방어하기
위해 애플리케이션의 매핑 로직(RowMapper 또는 JPA의 `DISTINCT` 키워드)은 이 중복된 Row들을 메모리 상에서 어떻게 단일 도메인 컬렉션으로 필터링 및 병합(Merge)해야 합니까?
