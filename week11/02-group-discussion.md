# `1. 사전 학습`

- Q. 잘못된 요청에 지금 내 코드는 어떻게 응답하고 있는가?
- A. 몇몇 처리된 코드 외엔 순수하게 예외를 발생시키고 전달
- Q. 에러 응답에 무엇을 담아야 사용자가 다음에 무엇을 할 수 있는가?
- A. 에러의 원인과 내용에 대한 정보
- Q. 변경·취소는 생성보다 어떤 에러 케이스가 더 많은가?
- A. 없는 값에 대한 요청에 따른 에러
- Q. 사이클1에서 만든 API 설계 규칙은 에러 상황에서도 그대로 유효한가?
- A. 개인적으로 추가한 규칙에 따르면 유효하다.

  > ◦ 동작하는 API 중 바람직한 API 가 좋은 API 다.
  >
  > - 올바른 요청에 올바른 응답을 보내는 것은 동작하는 API 다.
  > - **잘못된 요청에 합당한 응답을 보내는 것은 바람직한 API 다.**
  > - 명세만으로 목적/행위/결과를 추론할 수 있는 것은 바람직한 API 다.
  > - 드러나선 안 될 정보를 은닉하는 것은 바람직한 API 다.
  > - 엔드포인트는 `무엇`을, 메서드는 `어떻게`를 표현한다.

## **실험2: 예약 변경/취소 API 설계**

### **해볼 것**

- [x]  "사용자가 자신의 예약을 취소한다", "예약 날짜·시간을 변경한다"를 API로 만든다고 가정한다
- [x]  두 API의 URL·HTTP 메서드·요청·응답·상태 코드를 적는다
- [x]  변경/취소가 거부되어야 하는 케이스를 떠오르는 만큼 적고, 각 케이스의 응답을 설계한다

`사용자가 자신의 예약을 취소한다`

- 요청

```
DELETE localhost:8080/reservations/{id} HTTP/1.1
Content-Type: application/json
```

- 응답 (정상적 요청)

```
DELETE <http://localhost:8080/reservations/{id}>

HTTP/1.1 200

{
    "timestamp": "2026-05-11T16:22:04.102+00:00",
    "status": 200,
    "message": "예약이 정상적으로 취소되었습니다.",
    "path": "/reservations"
}
```

- 응답 (잘못된 요청)

```
DELETE <http://localhost:8080/reservations/{id}>

HTTP/1.1 400

{
    "timestamp": "2026-05-11T16:22:04.102+00:00",
    "status": 400,
    "message": "해당 예약 정보가 존재하지 않습니다.",
    "path": "/reservations"
}
```

`예약 날짜·시간을 변경한다`

- 요청

```
PUT localhost:8080/reservations/{id} HTTP/1.1
Content-Type: application/json

{
    "date": "2026-05-07",
    "name": "브라운",
    "timeId": 1,
    "themeId": 1
}
```

- 응답 (정상적 요청)

```
PUT <http://localhost:8080/reservations/{id}>

HTTP/1.1 200

{
    "timestamp": "2026-05-11T16:22:04.102+00:00",
    "status": 200,
    "message": "예약이 정상적으로 변경되었습니다.",
    "path": "/reservations",
    "reservation": {
        "id": {id},
        "name": "브라운",
        "date": "2026-05-07",
        "time": {
            "id": 1,
            "startAt": "09:00:00",
            "isAvailable": true
        },
        "theme": {
            "id": 1,
            "name": "폐병원 탈출",
            "description": "버려진 병원에서 벌어지는 기괴한 일들. 살아서 나가야 한다.",
            "thumbnailUrl": "<https://example.com/t1.jpg>"
        }
    }
}
```

- 응답 (잘못된 요청)

```
PUT <http://localhost:8080/reservations/{id}>

HTTP/1.1 400

{
    "timestamp": "2026-05-11T16:22:04.102+00:00",
    "status": 400,
    "message": "해당 예약 정보가 존재하지 않습니다.",
    "path": "/reservations"
}
```

`거부되어야 하는 케이스`

각각엔 타임스탬프가 포함된 것으로 가정

- 존재하지 않는 예약에 대한 수정/삭제
    - `400`, "message": "해당 예약 정보가 존재하지 않습니다."
- 다른 사용자의 예약에 대한 수정/삭제
    - `403`, "message": "자신의 예약 정보만 수정 및 삭제할 수 있습니다."
- 이미 지난 예약에 대한 수정/삭제
    - `400`, "message": "지나간 예약은 수정 및 삭제할 수 없습니다."
- 이미 존재하는 예약(동일한 날짜, 테마, 시간)으로의 수정

## 관찰할 것

### 어떤 요청이 500을 반환했는가?

이번 실험에서 500을 반환한 요청은 없었다.

- 존재하지 않는 예약 조회 시도: 405
- 필수 값 누락: 400
- 지난 날짜 예약 시도: 201
- 중복 예약 시도: 400
  500이 그대로 노출되지는 않았지만, 405 응답은 기본 에러 형식이라 사용자가 다음 행동을 판단하기 어렵다.

### 응답 본문에 무엇이 잘못됐는지가 들어 있는가?

요청마다 달랐다.

- 필수 값 누락이나 잘못된 값: 무엇이 빠졌는지, 어떤 값이 잘못됐는지 들어 있다.
- 중복 예약: 이미 예약된 시간이라는 원인이 들어 있다.
- 존재하지 않는 예약 조회 시도: `Method Not Allowed`만 있어서 정확히 무엇이 문제인지 알기 어렵다.
- 지난 날짜 예약: 잘못된 요청이라고 생각하지만 성공으로 처리된다. 그래서 문제 원인이 응답에 들어 있지 않다.

### 응답 본문에 다음에 뭘 할 수 있는지가 들어 있는가?

대체로 부족했다.

- 필수 값 누락은 빠진 값을 채우면 된다는 것을 알 수 있다.
- 시간 ID, 테마 ID 오류는 올바른 양수 값을 넣어야 한다는 것을 알 수 있다.
- 중복 예약은 다른 시간을 선택해야 한다는 것을 추측할 수 있지만, 가능한 시간은 알려주지 않는다.
- 405 응답은 사용자가 어떤 요청을 다시 보내야 하는지 알기 어렵다.
- 지난 날짜 예약은 성공 처리되기 때문에 사용자가 문제를 인식할 수 없다.

### 4개 응답의 형식이 일관적인가?

일관적이지 않다.

- 405 응답은 JSON 형식이다.
- 필수 값 누락과 중복 예약은 `[ERROR] ...` 문자열 형식이다.
- 지난 날짜 예약은 에러가 아니라 정상 생성 응답 JSON이다.
  따라서 현재 에러 응답은 형식이 일관되지 않다. 사용자가 응답을 보고 다음 행동을 정할 수 있게 하려면, 에러 응답 형식을 통일하고 "무엇이 잘못됐는지"와 "다음에 무엇을 할 수 있는지"를 담는 기준이 필요하다.

## 실험2. 예약 변경/취소 API 설계

### 1. 예약 취소 API 설계

- 목적: 사용자가 자신의 예약을 취소한다.
- HTTP Method: DELETE
- URL: DELETE /reservations/{id}
- 요청: 없음
- 성공 응답: 204 No Content
- 응답 본문: 없음
- 설계 이유: 예약을 취소하면 해당 예약을 더 이상 사용할 수 없게 되므로 DELETE를 사용한다. 요청 본문 없이 예약 ID만으로 취소 대상을 알 수 있고, 성공했을 때 추가로 내려줄 내용이 없으므로 204 No
  Content로 응답한다.

### 2. 예약 변경 API 설계

- 목적: 사용자가 예약 정보를 변경한다.
- HTTP Method: PUT
- URL: PUT /reservations/{id}
- 요청: {"name": "캐리", "date": "2026-05-20", "timeId": 2, "themeId": 1}
- 성공 응답: 200 OK
- 응답 본문: {"id": 1, "name": "캐리", "date": "2026-05-20", "time": {"id": 2, "startAt": "11:00:00"}, "theme": {"id": 1, "
  name": "시간조작자 연구소"}}
- 설계 이유: 예약 변경은 이름, 날짜, 시간, 테마를 한 번에 바꾸는 요청이라고 생각했다. 일부 값만 따로 바꾸면 헷갈리거나 오류가 생길 수 있으므로, 변경할 예약 정보를 한 번에 보내는 PUT이 적절하다고
  판단했다.

## 3. 변경/취소가 거부되어야 하는 케이스

### 3-1. 예약 변경 시 이미 예약된 날짜/시간인 경우

- 상황: 사용자가 예약 날짜나 시간을 변경하려고 했는데, 변경하려는 날짜와 시간에 이미 다른 예약이 있는 경우
- 상태 코드: 400 Bad Request
- 응답 본문: [ERROR] 해당 날짜/시간에 이미 예약이 있습니다.
- 사용자가 이걸 보면 뭘 할 수 있는가: 해당 날짜와 시간은 사용할 수 없다는 것을 알 수 있다. 사용자는 예약 가능 시간 조회 API를 다시 호출해서 다른 시간을 찾아볼 수 있다.
- 관찰: 이미 예약된 시간인지 여부는 서버가 판단해야 하는 비즈니스 규칙이다. 현재 프로젝트에서는 잘못된 요청을 400으로 응답하는 흐름이 있으므로, 변경 요청에서도 400으로 통일한다.

## 4. 관찰할 것

### 변경 API에 어떤 HTTP Method를 쓸 것인가? 무엇을 기준으로 결정했는가?

변경 API에는 PUT을 사용한다.
이유는 예약 정보를 한 번에 변경하는 방식이 더 명확하다고 생각했기 때문이다. 이름, 날짜, 시간, 테마를 각각 따로 바꾸게 하면 어떤 값은 바뀌고 어떤 값은 그대로인지 헷갈릴 수 있다. 그래서 변경 요청에서는 예약
정보를 한 번에 보내고, 서버는 그 정보로 예약을 변경하도록 설계했다.

### 이미 지난 예약을 취소하는 요청은 허용할 것인가?

허용한다.
처음에는 지난 예약 취소를 거부할 수도 있다고 생각했지만, 관리자 측 실수로 환불을 해줘야 하는 경우가 있을 수 있다. 그런 경우에는 이미 지난 예약이라도 취소하거나 삭제할 수 있어야 한다고 생각했다.
따라서 지난 예약 취소는 거부하지 않고, 정상적으로 처리한다.

- 요청: DELETE /reservations/{id}
- 성공 상태 코드: 204 No Content
- 응답 본문: 없음

### 변경하려는 시간이 이미 차 있을 때, 예약 가능한 다른 시간까지 알려줄 것인가?

알려주지 않는다.
이미 예약된 시간이라는 사실만 알려주면 충분하다고 생각했다. 사용자는 필요하면 예약 가능 시간 조회 API를 다시 호출해서 가능한 시간을 확인할 수 있다. 변경 API의 역할은 변경 요청을 처리하거나 거부하는
것이고, 가능한 시간 목록까지 함께 알려주는 것은 변경 API의 책임이 아니라고 생각했다.

- 거부해야 하는 케이스: 예약을 변경하려는 날짜와 시간에 이미 다른 예약이 있는 경우
- 상태 코드: 400 Bad Request
- 응답 본문: [ERROR] 해당 날짜/시간에 이미 예약이 있습니다.
- 그 응답을 받은 사용자가 다음에 할 수 있는 행동: 사용자는 해당 날짜와 시간으로는 예약을 변경할 수 없다는 것을 알 수 있다. 그래서 예약 가능 시간 조회 API를 다시 호출하거나, 다른 날짜와 시간을 선택해서
  다시 변경 요청을 보낼 수 있다.

## 토론 활동에서 말할 근거

이번 실험을 하면서 에러 응답에는 최소한 에러 코드와 메시지가 필요하다고 생각했다. 사용자가 응답을 봤을 때 어떤 에러인지 알 수 있어야 대응할 수 있기 때문이다.
또 에러 응답도 JSON 형식으로 내려주는 것이 좋다고 생각한다. 문자열보다 JSON이 프론트엔드나 클라이언트에서 처리하기 편하기 때문이다.
실제로 필수 값 누락이나 중복 예약은 `[ERROR] ...` 메시지가 있어서 무엇이 문제인지 어느 정도 알 수 있었다. 하지만 문자열 응답이라 클라이언트에서 에러 코드, 메시지 같은 값을 구분해서 쓰기 어렵다고
느꼈다.
반대로 405 응답은 JSON 형식이지만, 예약이 없는 것인지, API가 없는 것인지, 요청 방식이 잘못된 것인지 메시지가 부족해서 헷갈렸다.
지난 날짜 예약은 비즈니스 정책상 거부되어야 한다고 생각하지만 201 Created로 성공 처리되었다.

삭제 API도 고민이 되었다. 나는 취소 성공 시 204 No Content로 설계했지만, 정말 아무 내용도 주지 않는 것이 맞는지 궁금하다.
예를 들어 삭제된 예약 정보나 삭제된 개수 같은 것을 반환해야 사용자가 더 명확하게 알 수 있지 않을까 하는 생각도 들었다.

변경 API는 PUT을 쓰는 쪽으로 생각했다.
예약 정보를 일부만 바꾸기보다 이름, 날짜, 시간, 테마를 한 번에 보내는 방식이 더 명확하다고 느꼈기 때문이다. 다만 PUT과 PATCH의 차이는 더 공부해서 토론해보고 싶다.

## 토론에서 가장 묻고 싶은 질문 1개

에러 응답에는 "무엇이 잘못됐는지"만 담아도 충분한가, 아니면 사용자가 다음에 무엇을 해야 하는지까지 대략적으로 알려줘야 하는가?

- 나는 에러 응답을 보고 사용자가 대응할 수 있는지까지 생각해서 응답 메시지를 줘야 한다고 생각한다. 사용자가 대응할 수 있는지에 대한 기준에 대해도 토론하고싶다.

추가로 PUT과 PATCH 중 예약 변경에 어떤 메서드를 쓰는 것이 더 적절한지도 토론해보고 싶다. 나는 현재는 PUT 쪽이 더 맞다고 생각하지만, 그 이유를 더 분명하게 정리하고 싶다.

## 예약 변경과 취소

patch란?

- 리소스 부분을 수정한다.
- 멱등성이 아닐 수 있다.
- 변경할 필드만 포함한다.
- 일부 필드만 수정할 때, 사용한다. (대부분의 경우가 이렇다.)

put이란?

- 리소스 전체를 수정한다.
- 멱등성이 보장된다.
- 리소스 전체를 포함한다.
- 설정 파일 처럼, 전체 저장처럼 완전한 대체가 명확한 상황에서 사용한다.

### 1. 사용자가 자신의 예약 취소 API

사용자의 예약의 상태를 취소 상태로 변경한다.

이때 그러면 예약에 상태라는 필드가 추가되어야 한다.

예약 상태 (대기, 확정, 완료, 취소)

요청

```java
PATCH http://localhost:8080/reservations/{reservationId}
        {
        "status":"CANCELLED"
        }
```

```java
PATCH http://localhost:8080/reservations/{reservationId}/cancel
```

### 2. 예약 날짜·시간 변경 API

예약의 날짜, 시간을 변경한다.

존재하는 필드에 대해서만 변경한다.

요청

```java
PATCH http://localhost:8080/reservations/{reservationId}
        {
        "date":"2026-05-15",
        "timeId":1
        }
```

변경/취소가**거부되어야 하는 케이스**를 떠오르는 만큼 적고, 각 케이스의 응답을 설계한다

공통

- 예약이 존재하지 않는 경우
    - 404 Not Found
- 자신의 예약이 아닌 경우
    - 403 Forbidden
- 이미 취소가 된 경우
    - 409 Conflict
- 이미 완료된 예약은 취소 불가
    - 409 Conflict

`취소`

- 과거 날짜의 예약은 취소 불가
    - 422 Unprocessable Entity

`변경`

- 변경할 날짜가 과거 날짜일 경우
    - 400 Bad Request
- 존재하지 않는 시간Id일 경우
    - 404 Not Found
- 이미 변경할 날짜, 시간이 예약된 경우
    - 409 Conflict

`400`  Bad Request 형식의 오류

`422`  Unprocessable Entity 비즈니스의 규칙 위반

---

- `변경 API에 어떤 HTTP Method를 쓸 것인가? 무엇을 기준으로 결정했는가?`

변경은 PATCH 를 사용할 거 같다.

PUT의 경우 전체를 대체할 수 있어야하지만,

PATCH의 경우 존재하는 필드에 대해서만 수정하는게 맞을거 같다.

- `"이미 지난 예약을 취소"는 허용할 것인가, 거부할 것인가? 거부한다면 상태 코드는?`

이미 날짜 지난 예약 취소는 거부한다.

상태코드는 ? 422 Unprocessable Entity

- `변경하려는 시간이 이미 차 있을 때, 응답에 "예약 가능한 다른 시간"까지 알려줄 것인가?`

알려주지 않을것이다.

왜냐하면 예약 가능한 다른 시간을 다시 API 조회로 해결하면 되기 때문이다.

#### **1. 리소스 식별 기준**

> **URL을 정할 때 무엇을 리소스로 보고, 무엇을 필터·표현 조건으로 보는가**
>

`Rule 1.` 도메인 분석을 통해, 실행 흐름에 나타나는 명사를 리소스로 식별한다.

리소스 식별 규칙 자체는 추상적으로 설정하되,

API 명세 설계에 대한 각자의 관점에 맞춰 판단하고

- 서버/도메인 모델 관점
- 클라이언트/멘탈 모델 관점

이후 피드백 활동에서 각자의 경험을 공유해 규칙을 구체화한다.

#### **2. 서버/클라이언트 책임 기준**

> **어떤 판단을 서버가 책임지고, 어떤 판단을 클라이언트가 책임지는가**
>

- **(If-Then)** 만약 판단에 필요한 원천 데이터가 서버에만 존재한다면 → **해당 로직은 서버가 책임지고 결과값을 내려준다.**
    - 예: "예약 가능 여부(`available`)"는 서버가 기존 예약 현황을 계산하여 응답에 포함한다.
- **(If-Then)** 만약 클라이언트가 받은 데이터를 가공 없이 화면에 그리는 역할이라면 → **서버는 클라이언트가 바로 사용할 수 있는 형태(UI friendly)로 응답을 설계한다.**
- **(우선순위)** 책임 판단 순서:

1. 데이터의 무결성을 보장해야 하는 곳이 어디인가? (보통 서버)
2. 클라이언트가 중복된 계산 로직을 가질 필요가 있는가?

- **(금지)** 클라이언트에게 비즈니스 로직 판단을 떠넘기지 않는다.
    - **이유:** 데이터 불일치 문제를 방지하고 클라이언트의 복잡도를 낮추기 위함이다.

#### **3. 관리자/사용자 API 분리 기준**

> **같은 자원을 다루는 관리자와 사용자 API를 분리하는가, 합치는가**
>

- **(If-Then)** 만약 리소스의 도메인 모델과 생성 행위가 동일하다면 → **관리자와 사용자 구분 없이 같은 엔드포인트를 사용한다.**
    - 예: `POST /reservations` (누가 생성하든 '예약 생성'이라는 행위는 동일함)
- **(If-Then)** 만약 사용자의 권한에 따라 응답 필드의 구성이나 노출 데이터가 현저히 다르다면 → **도메인 목적에 따라 API를 분리하거나 권한 로직으로 분기한다.**
- **(우선순위)** 결정 순서:

1. 리소스의 식별자가 동일한가?
2. 조회/수정의 목적과 Context가 동일한가?

#### **4. 우리 그룹의 "좋은 API" 정의**

> **동작하는 것을 넘어, 어떤 조건을 만족해야 좋은 API라고 부를 것인가**
>

좋은 API

- **정의:** 동작을 완벽히 보장하는 것은 물론, **뛰어난 가독성**을 바탕으로 명세만 보아도 요청과 행동을 직관적으로 식별할 수 있는 API.
- **특징:**
    - **직관성:** URL(리소스 명사)과 HTTP 메서드(행위 동사)의 조합만으로 해당 API가 어떤 자원에 대해 무슨 작업을 수행하는지 누구나 명확히 추론할 수 있습니다.
    - **명확성:** 리소스의 정의, 메서드 시그니처 등에 대해 팀이 합의한 명확한 기준과 규칙을 엄격하게 따릅니다.
    - **문서화의 역할**: 잘 짜인 API 명세 자체가 훌륭한 설명서 역할을 하여, 개발자 간의 소통 비용과 유지보수 비용을 크게 낮춥니다.
