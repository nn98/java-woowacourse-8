# `4. 피드백 활동`

1. ### 상태 위치를 결정할 때 가장 고민된 것은?
- 답변 : 그 상태를 보유할 책임과, 상태를 보유 여부에 따른 객체의 유형/특성
- 이유 : 토론 활동에서 고민한 내용을 기반으로, 상태의 특성에 따라 그 책임을 결정할 수도,  
  상태의 분리를 통해 불변/가변 객체를 만들 수도 있음을 확인했기에 책임과 불변성을 고민의 지표로 삼음

1. ### 불변 객체를 써서 좋았던 점은?
- 답변 : 불변성 + 검증을을 통한 유효성 을 믿고 과감하게 사용 가능 
- 이유 : 도메인 간 의존에서도 포장이 필요한 이유는 조작에 대한 방어인데 불변 객체가 조작될 가능성이 없기에 
  
1. ### 블랙잭 때와 상태 관리 방식이 어떻게 다른가?
- 답변 : 보다 적은 종류, 보다 높은 복잡도 
- 이유 : 아직 상태 패턴을 적용하거나 하지 않아 명확하진 않지만 답변과 같이 생각함 

1. ### 다형성을 적용해서 좋았던 점은?
- 답변 : 중복 코드와 과도한 분기 제거 
- 이유 : 다형성 호출이라는 처음엔 의문스러워도 나중엔 감탄스러운 기능을 활용해  
  동일한 호출, 다른 기능이라는 결과를 만들어 낼 수 있음 

1. ### 다형성이 과했던 부분은 없는가?
- 답변 : 오히려 아직 부족하지 않나 생각
- 이유 : 코드가 아직 갈길이 멀기에..

1. ### 조건문이 언제 문제가 된다고 생각하는가?
- 답변 : 과도한 조건 분기를 통한 가독성, 유지보수성, 성능 저하 
- 이유 : `누구누구의 법칙` 을 참조하면 사람은 7개 이상의 내용을 동시에 기억할 수 없는데  
  그보다 많은 조건을 어떻게 보는 것 만으로 예상하고 수정하고 활용할 수 있을까?  
  실행하는 주체는 컴퓨터일지라도 그걸 만들고 수정하고 사용할 주체는 인간이다.  
  컴퓨터는 그저 0 과 1 만으로 모든 일을 수행할 수 있지만  
  인간은 0 과 1 말고도 너무 많은 것을 아는 멍청이기에 더욱 직관적이고 간결해야 한다  
  성능은 잘은 모르지만 if - else 같이 분기할 때마다 CPU 성능이 꽤 많이 소모된다고 한다.
