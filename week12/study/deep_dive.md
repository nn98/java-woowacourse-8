### 1. 의존성과 빌드 도구

**질문**
'의존성(dependency)을 추가한다'는 것은 무슨 의미인가?
'build.gradle'에 새로운 의존성을 한 줄 추가하면 어떤 일이 발생하는가?

**답변**
프로젝트가 의존(참조)하는 라이브러리를 추가한다.
해당 라이브러리를 다운로드하고 런타임에 등록한다.

**수정**
부분 정답입니다. 라이브러 참조는 맞으나 동작 과정의 구체성이 부족합니다.
의존성 추가 시 Gradle이 원격 저장소에서 `.jar` 파일을 로컬 캐시로 다운로드하고, 이를 '컴파일 클래스패스'와 '런타임 클래스패스'에 각각 알맞게 주입하여 컴파일과 실행이 모두 가능하도록 구성합니다.

<br>

### 2. 라이브러리와 웹 프레임워크

**질문**
library, web framework 를 모르는 사람에게 뭐라고 설명하면 이해하기 좋을까?

**답변**
필요한 기능들을 이미 정형화된 상태로 모아놓은 꾸러미(라이브러리).
인터넷 프로그램을 개발하기 위한 도구상자이자 정형화된 틀(프레임워크).

**수정**
목적에 대한 설명은 타당하나 기술적인 차이점 구분 기준이 누락되었습니다.
핵심 기준은 '제어 흐름의 주도권(제어의 역전, IoC)'입니다. 라이브러리는 내 코드가 주도권을 쥐고 외부 코드를 '호출'해서 사용하는 것이며, 프레임워크는 외부 코드가 주도권을 쥐고 내 코드를 '호출당하게'
만드는 것입니다.

### 3. 호스트명과 포트

**질문**
localhost 와 port : 8080 의 의미는?

**답변**
실행되는 컴퓨터의 ip 주소. 특정 주소의 포트 중 8080번 포트.

**수정**
오답입니다.
localhost는 주소 자체가 아니라, `127.0.0.1`이라는 루프백 네트워크 IP 주소를 가리키는 호스트명(Domain Name)입니다.
Port는 특정 주소의 하위 개념이 아니라, 하나의 운영체제 내에서 실행 중인 여러 프로세스 중 목적지 프로세스를 정확히 찾아가기 위한 '논리적 통신 종단점(Endpoint)' 식별자입니다.

**추가 학습**
프론트엔드와 백엔드의 포트(Origin)가 다를 때 브라우저가 통신을 차단하는 것을 CORS 정책이라고 합니다. 이를 인프라 단에서 리버스 프록시(Nginx)로 통일해 우회할 수도 있지만, 스프링 내부의 글로벌
정책으로 해결하려면 `WebMvcConfigurer` 인터페이스를 구현해 `addCorsMappings` 메서드를 오버라이딩하는 것이 정석입니다.

### 4. 스프링 부트 실행 로그

**질문**
springboot 실행 시 보이는 `INFO ... [ Test worker ] ... TomcatWebServer : Tomcat initialized with port 8080 (http)` 로그 내용을
이해하고 있는가?

**답변**
스프링 내장 톰캣 서버가 8080포트에 초기화(실행)됨.

**수정**
부분 정답입니다. 현상에 대한 해석은 맞지만 `[ Test worker ]` 스레드명 분석이 빠졌습니다.
이는 실제 애플리케이션 기동이 아니라, 테스트 실행 컨텍스트 내부에서 내장 톰캣이 띄워졌음을 의미하는 결정적 단서입니다.

**추가 학습**
기본 내장 톰캣을 다른 WAS(Undertow 등)로 교체하려면, `build.gradle`의 `spring-boot-starter-web` 의존성에서 톰캣을 명시적으로 `exclude`(제외) 시키고 원하는 WAS
의존성을 새로 추가하면 됩니다.

### 5. Gradle 기초

**질문**
gradle에서 제공하는 build와 clean은 무엇인가?
implementation, runtimeOnly, testImplementation 는 뭐가 다른 걸까?

**답변**
빌드는 현재 프로젝트의 코드를 빌드, 클린은 빌드된 내용 클리어. 스코프의 차이는 모름.

**수정**
클린(clean)은 이전 빌드 잔재(build 또는 out 디렉터리)를 완전히 삭제하여 새로운 빌드 결과물에 과거 데이터가 영향을 주지 않도록 완벽히 격리하는 작업입니다.

**추가 학습**
소스코드(`.java`)를 바이트코드(`.class`)로 변환하는 단일 작업이 컴파일(`compileJava`)이며, 최종 배포 산출물(`.jar`)을 만드는 전체 생명주기를 빌드(build)라고 합니다.
의존성 스코프 중 `implementation`은 컴파일과 실행 모두에, `runtimeOnly`는 실행 시점에만(예: H2 DB 드라이버), `testImplementation`은 테스트 시점에만 사용됩니다.

### 6. JSON과 직렬화

**질문**
json 이란 무엇이며 직렬화, 역직렬화는 무엇인가? 왜 필요한가?

**답변**
자바스크립트 표현 언어. 자연어와 기계어 간의 변환. 더 효율적으로 전달하기 위해.

**수정**
오답입니다.
JSON은 언어가 아니라 자바스크립트 객체 문법으로 구조화된 데이터를 표현하기 위한 텍스트 기반의 데이터 교환 표준 포맷입니다.
직렬화/역직렬화는 기계어 변환이 아닙니다. 메모리 상의 '객체'를 네트워크 전송이나 DB 저장에 적합한 '바이트 스트림이나 문자열(JSON)'로 변환하는 과정이 직렬화이며, 그 반대가 역직렬화입니다.

**추가 학습**
스프링 부트 Web 모듈에는 Jackson이라는 직렬화/역직렬화 엔진(`ObjectMapper`)이 내장되어 있습니다. 이 엔진이 리플렉션을 통해 JSON을 DTO 객체로 조립하려면, 클래스 내부에 파라미터가 없는
기본 생성자(`@NoArgsConstructor`)가 반드시 존재해야 합니다.

### 7. XHR과 API

**질문**
XHR 이란?

**답변**
모름.

**수정**
XHR(XMLHttpRequest)은 브라우저가 제공하는 API로, 페이지 새로고침 없이 웹 서버와 비동기적으로 데이터를 교환(AJAX)하는 현대 동적 UI의 핵심 기술입니다. (현대 표준 내장 비동기 API는
fetch이며, axios는 외부 라이브러리입니다.)
API(Application Programming Interface)는 소프트웨어 시스템끼리 데이터를 주고받기 위해 정해놓은 규격이자 계약서입니다.

### 8. 웹 서버 vs WAS

**질문**
웹서버 vs 웹 애플리케이션 서버의 차이는?

**답변**
웹서버는 통신의 측면, 웹앱서버는 기능/서비스의 측면.

**수정**
방향성은 정답입니다.
웹 서버(Nginx, Apache)는 정적 컨텐츠 제공 및 클라이언트와의 직접 통신(리버스 프록시)을 담당합니다. 반면 WAS(Tomcat)는 DB 조회 및 비즈니스 로직 연산이 필요한 동적 컨텐츠 생성(서블릿
컨테이너 역할)을 담당합니다.

**추가 학습**
트래픽이 몰릴 때 앞단에 위치한 Nginx 등의 웹 서버가 여러 대의 WAS 인스턴스로 요청을 골고루 분산시켜 시스템 부하를 조절하는 기술을 로드 밸런싱(Load Balancing)이라고 합니다.

### 9. 클린코드와 평생 학습

**질문**
내가 클린코드를 지향하는 목적은 무엇인가? 개발자는 '평생 학습해야 한다' 라는 말에 동의하는가?

**답변**
효과적인 유지보수 및 소통, 비용절감. 끊임없는 분야의 변화와 도태 방지, 통찰력 향상을 위해 동의함.

**수정**
완벽한 통찰이자 정답입니다.
코드는 작성되는 시간보다 읽히고 수정되는 시간이 압도적으로 깁니다. 가독성과 유지보수성이 떨어지면 기능 추가에 드는 시간과 비용(기술 부채, Technical Debt)이 기하급수적으로 증가합니다. 기술 스택의
휘발성을 인지하고 CS나 아키텍처 같은 근본 원리를 중시하는 태도는 개발자로서 매우 훌륭합니다.

**추가 학습**
외부 동작(API 응답 결과, 기능)은 전혀 바꾸지 않고 내부 구조와 가독성만 개선하는 작업을 리팩토링(Refactoring)이라고 합니다. 외부 동작이 변하지 않음을 증명해야 하므로, 기존에 작성된 테스트 코드가
깨지지 않고 모두 통과해야만 올바른 리팩토링이 성립합니다.
