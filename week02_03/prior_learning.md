# `1. 사전 학습`

- Q. 애플리케이션을 직접 실행해서 검증하는 것으로 충분한가?
- A. 충분하지 않다. 실행 과정에서 모든 기능을 직접 검증하는 것은 비효율적.  
  또한 누락 위험성과 유지보수성 저하라는 위험 또한 존재.
- Q. 테스트를 통해 어디까지 검증할 것인가?
- A. 유효성은 자체 생성 과정에서, 단위 테스트는 각 단위별 유효한 기능 검증  
  통합 테스트는 각 단위의 유기적 관계와 그 기능에 대해 검증
- Q. 테스트를 하기 어려운 코드는 무엇인가?
- A. 의존성이 복잡하게 연결되고 기능과 책임이 병존하는 코드
- Q. 무엇을 테스트하고 무엇을 버릴 것인가?
- A. 지금 상황에서 모든 기능과 상황에 대한 100% 커버리지를 목표로 하기보단  
  가장 작은 단위부터 순차적으로, 또 점진적으로 테스트를 늘려나가기  
  Keep - 개체 유효성 / 기능 유효성  
  Away - 통합 테스트 / 시나리오 테스

### **기록하기**

> **형식**: 각 관찰은 **사실 1개 + 문제 1개**

```
예시:
- 사실: 케이스를 6개로 늘리니 main에서 3곳을 수정해야 했다
- 문제: 하나를 빼먹고 수정해서 오류를 놓칠 뻔했다
```

> ### 케이스 증가 시
> - `사실`: 케이스의 수가 변화할 때 마다 main의 반복 조건문을 수정
> - `문제`: 수동적으로 코드를 수정해야 원하는 방식의 규칙을 구현 및 테스트 가능
>
> ### 수정 중 놓친 사항
> - `사실`: 실험 과정에선 놓친 사항 X. 케이스 수에 맞춰 반복문의 조건만 수정
> - `문제`: 하지만 기능 구현 과정에서 메서드 분리 등 코드의 복잡도가 증가할 경우, 누락 사항이 발생할 가능성이 급격히 증가
>
> ### 이전 케이스 동작 확인
> - `사실`: 실험 과정에선 정상 동작 확인. 동일 케이스 시도 후 추가된 만큼의 케이스 시도하는 방식
> - `문제`: 케이스가 늘어날수록 효율 저하. 비효율적으로 앞선 케이스를 반복하지 않을 경우 이전 케이스 동작을 보장 불가

### **해볼 것**

  - [x] 경계 케이스나 예외 케이스 1개 추가

### **관찰할 것**

- 테스트를 작성하기 어려운 코드가 있는가?
- 어렵다면, 무엇 때문에 어려운가?
- 테스트를 쉽게 하려면 코드를 어떻게 바꾸고 싶은가?

### **기록하기**

관찰 기록에 추가:

- 테스트가 어려웠던 코드 1곳
- 바꾸고 싶은 부분과 이유

> ### 테스트를 작성하기 어려운 코드가 있는가?
> 
> ```java
> @DisplayName("정상적 숫자/문자로 카드 생성")
> @ParameterizedTest
> @ValueSource(strings = {"2", "4", "7", "9", "Q"})
> void 정상적_숫자_문자로_카드_생성(String denomination) {
>      assertThatNoException()
>          .isThrownBy(() -> Card.findByDenomination(denomination));
> }
> ```
> 
> - `사실` 
>   - 테스트 자체는 위와 같이 NoException으로 정상 동작 검증
> - `문제` 
>   - 생성 과정에서 예외 발생 유무는 검증이 가능.  
>   - 하지만 예외 없이 생성된 각 개체가 내가 의도한 대로 생성되었는지,  
>     그 내부에 대한 유효성 검증은 전무.
> - `예시`
>   - 생성된 카드들로 점수 계산 시 예상한 결과가 나오지만 (2, 4, 7, 9, Q) → 32,  
>     이 각각의 카드가 정말 2, 4, 7, 9, Q(10)이란 값을 가지고 있는 게 맞을까?
>
> ### 어렵다면, 무엇 때문에 어려운가? 
> - `사실`
>   - 현재 예외 발생의 유무로만 검증을 진행
> - `문제`
>   - 정상 동작 여부 뿐만 아니라  
>     동작의 결과로 생성된 객체의 유효성을 검증해야 함
>
> ### 테스트를 쉽게 하려면 코드를 어떻게 바꾸고 싶은가? 
> - `사실`
>   - 동작(행위) 자체의 유효성 검증은 당연하며 그 결과에 대한 유효성 검증도 필요
> - `문제`
>   - 결과를 확인하기 위해 getter/setter 구현 시 결합도 상승,  
>     단순 테스트용 equals 구현 시 도메인 로직과의 무관성 우려
> - `해결`
>   - 단순히 값을 저장만 하는 클래스가 아닌 VO(Value Object)로 기능을 확대.
> - `근거`
>   - 해당 값의 유효성을 확인하고 보유하는 로직은   
>     도메인에 '무관하지 않은' 필수 로직이며,  
>     이를 통해 TDA(Tell, Don't Ask) 원칙 준수 및  
>     객체 간 책임 분리를 통한 명확한 검증 구현 가능

## 산출물(제출 내용)
- ### Q. 테스트 단위에 대한 개인적인 판단과 그 근거를 작성해 주세요.
- ### A.
  - 핵심은 최소 단위, 점진적 확대
      - 가장 작은 기능 단위로 테스트하되, 서비스에서 명확히 사용되는 기능에 한정
      - 상위 기능이 하위 기능의 조합이라면, 가장 낮은 단위의 기본적 기능부터 검증해야 함

- ### Q. 반드시 테스트해야 한다고 판단한 항목과 그 이유를 작성해 주세요.
- ### A.
  - 최소 단위의 **`도메인 객체의 핵심 비즈니스 로직과 상태 변경`**
      - 베이스 기능이 명확히 동작해야 상위 기능의 정상적 동작을 보장
      - 테스트와 설계 학습이라는 목표를 고려할 때, 가장 직관적이고 직접적인 검증이 가능한 최소 단위 테스트가 바람직함

- ### Q. 테스트하지 않아도 된다고 판단한 항목과 그 이유를 작성해 주세요.
- ### A.
  - IO 로직 및 UI 인터페이스
      - IO 단의 기본 동작은 자바 표준 라이브러리가 처리하며,  
        이는 핵심 비즈니스 로직과 연관되지 않는 외부 영역임
      - 입출력 로직 이하의 기능적/구조적 동작은 
        도메인 객체의 최소 단위 테스트에서 이미 검증되었어야 함
  - 실제 비즈니스 로직에 적용되지 않는 기능
      - 사용되지 않는 기능은 존재 목적을 판별하여 제거하거나  
        서비스에 활용하도록 관리 (`YAGNI` 원칙 준수)
