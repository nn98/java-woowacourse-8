    - 메서드의 수가 많을수록 SRP 위반

- # [`상속과 조합`]()
  - `상속`
  - 상속은 코드 재사용을 위한 강력한 수단, 항상 최선은 아니다.
  - 객체 사이의 관계를 복잡화하지만 그 근본 원인은 상속 자체가 아니다.
  - 완전한 배제가 아닌 올바른 사용을 목적으로 한다.
  - ```java
    public class Document {
        public int length() {
            return this.content().length;
        }
        
        public byte[] content() {
            // 문서의 내용을
            // 바이트 배열로 로드한다
        }
    }
        
    public class EncryptedDocument extends Document {
        @Override
        public byte[] content() {
            // 문서를 로드해서,
            // 즉시 복호화하고,
            // 복호화한 내용을 반환한다.
        }
    }
    ```
  - 위 코드에서, `length()` 호출 시, 자식 클래스의 `content()` 호출 발생.
  - 자식 클래스 오버라이딩이 부모 클래스 메서드 성능/방식을 오염시키는 사이드 이펙트
  - `상속을 위한 설계와 문서를 왁변히 갖추거나,`
  - `그럴 자신이 없다면 상속을 아예 금지하라.`
  - 클래스의 `final`
  - 메서드의 `abstarct`
    - 자식에게 책임을 명확히 이관.
    - 오버라이딩이 필연적이니, 함부로 호출(Self-use) 하지 않도록 명시
  - `상속` 은 부모와 자식 간의 결합도를 극히 높이기에 발생.
  - 상속 대신 `조합` 패턴으로 구현하는 것이 권장됨.
  - `조합`
  - 기존 클래스를 `상속`해 클래스를 확장하는 대신,
  - 신규 클래스에 필드로 기존 클래스 인스턴스를 참조하게.
- `확장과 정제`
  - 상속이 적절한 경우는 `확장` 하는 것이 아닌 `정제` 하는 경우.
  - `확장`
    - 새로운 행동을 덧붙여 기존의 행동을 부분적으로 보완
  - `정제`
    - 부분적으로 불완전한 행동을 완전하게 만듦
  - 중요성의 우선순위 변동 `재사용성` -> `유연성`
---
- #### `가변/불변 객체`
  - 내부 필드를 `final`,
  - 조회엔 복사본을,
  - 조작엔 신규 불변 복사본을.
  - `식별자 변경 문제` 차단
  - 참조한 원본을 변경하게 되면,
  - 그 참조를 보유한 모든 객체의 해당 값이 변경된다.
  - 과연 그것이 올바른 상황인가?
    - 불변 객체에선 실별자 변경 문제 방지.
  - `실패 원자성` 보유
    - 불변 객체는 완전하고 견고한 상태이거나 / 아예 실패해야 한다
  - `시간적 결합` 제거
    - 불변 객체는 항상 그 값을 유지하고,
    - 변경 등의 조작엔 신규 불변 복사본을 제공한다.
    - 이에 시간적 결합, 즉 호출 위치에 따른 가변적인 값이 아닌
    - 항상 초기의 값을 유지한다는 불변성을 보장한다.
  - `스레드 안전성` 보유
    - 객체가 여러 스레드에서 동시에 사용될 수 있고, 예측 가능한 결과를 보장
  - `단순성` 보유
    - 객체가 더 단순해질수록 응집도는 높아지고, 유지보수는 쉬워진다.
    - 불변 객체는 생성자 안에서만 상태를 초기화 할 수 있기에, 크기가 작아진다.
  - | 특성	          | 효과                |
        |--------------|-------------------|
    | 식별자 변경 문제 없음 | 컬렉션, 맵에서 안전하게 사용  |
    | 실패 원자성       | 중간 상태로 깨지지 않음     |
    | 시간적 결합 없음    | 코드 순서에 의존하지 않음    |
    | 스레드 안전성      | 동시성 버그 없음         |
    | 단순성          | 이해하기 쉽고 유지보수하기 쉬움 |
