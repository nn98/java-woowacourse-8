- # [`상속과 조합`]()
  - `상속`
  - 상속은 코드 재사용을 위한 강력한 수단, 항상 최선은 아니다.
  - 객체 사이의 관계를 복잡화하지만 그 근본 원인은 상속 자체가 아니다.
  - 완전한 배제가 아닌 올바른 사용을 목적으로 한다.
  - ```java
    public class Document {
        public int length() {
            return this.content().length;
        }
        
        public byte[] content() {
            // 문서의 내용을
            // 바이트 배열로 로드한다
        }
    }
        
    public class EncryptedDocument extends Document {
        @Override
        public byte[] content() {
            // 문서를 로드해서,
            // 즉시 복호화하고,
            // 복호화한 내용을 반환한다.
        }
    }
    ```
  - 위 코드에서, `length()` 호출 시, 자식 클래스의 `content()` 호출 발생.
  - 자식 클래스 오버라이딩이 부모 클래스 메서드 성능/방식을 오염시키는 사이드 이펙트
  - `상속을 위한 설계와 문서를 왁변히 갖추거나,`
  - `그럴 자신이 없다면 상속을 아예 금지하라.`
  - 클래스의 `final`
  - 메서드의 `abstarct`
    - 자식에게 책임을 명확히 이관.
    - 오버라이딩이 필연적이니, 함부로 호출(Self-use) 하지 않도록 명시
  - `상속` 은 부모와 자식 간의 결합도를 극히 높이기에 발생.
  - 상속 대신 `조합` 패턴으로 구현하는 것이 권장됨.
  - `조합`
  - 기존 클래스를 `상속`해 클래스를 확장하는 대신,
  - 신규 클래스에 필드로 기존 클래스 인스턴스를 참조하게.
- `확장과 정제`
  - 상속이 적절한 경우는 `확장` 하는 것이 아닌 `정제` 하는 경우.
  - `확장`
    - 새로운 행동을 덧붙여 기존의 행동을 부분적으로 보완
  - `정제`
    - 부분적으로 불완전한 행동을 완전하게 만듦
  - 중요성의 우선순위 변동 `재사용성` -> `유연성`
---
- #### `가변/불변 객체`
  - 내부 필드를 `final`,
  - 조회엔 복사본을,
  - 조작엔 신규 불변 복사본을.
  - `식별자 변경 문제` 차단
  - 참조한 원본을 변경하게 되면,
  - 그 참조를 보유한 모든 객체의 해당 값이 변경된다.
  - 과연 그것이 올바른 상황인가?
    - 불변 객체에선 실별자 변경 문제 방지.
  - `실패 원자성` 보유
    - 불변 객체는 완전하고 견고한 상태이거나 / 아예 실패해야 한다
  - `시간적 결합` 제거
    - 불변 객체는 항상 그 값을 유지하고,
    - 변경 등의 조작엔 신규 불변 복사본을 제공한다.
    - 이에 시간적 결합, 즉 호출 위치에 따른 가변적인 값이 아닌
    - 항상 초기의 값을 유지한다는 불변성을 보장한다.
  - `스레드 안전성` 보유
    - 객체가 여러 스레드에서 동시에 사용될 수 있고, 예측 가능한 결과를 보장
  - `단순성` 보유
    - 객체가 더 단순해질수록 응집도는 높아지고, 유지보수는 쉬워진다.
    - 불변 객체는 생성자 안에서만 상태를 초기화 할 수 있기에, 크기가 작아진다.
  - | 특성	          | 효과                |
            |--------------|-------------------|
    | 식별자 변경 문제 없음 | 컬렉션, 맵에서 안전하게 사용  |
    | 실패 원자성       | 중간 상태로 깨지지 않음     |
    | 시간적 결합 없음    | 코드 순서에 의존하지 않음    |
    | 스레드 안전성      | 동시성 버그 없음         |
    | 단순성          | 이해하기 쉽고 유지보수하기 쉬움 |

# 상속과 조합
- 완전 상속일 때(IS-A) : 클래스 상속
  - 부모의 내부 구현과 강한 결합, 캡슐화 파괴
- 부분 상속일 때(HAS-A) : 클래스 필드
  - 캡슐화 유지, 결합도 감소
- `상속보다는 조합을 사용하라`
- `조합 패턴`

### 방식
- 기존의 클래스를 확장(extends) 하는 대신 
- 기존의 클래스를 인스턴스로 새로운 클래스가 참조
- 참조한 클래스의 기능에 대한 책임은 참조한 클래스로 이관
- 100% 완벽한 상하 관계가 아닌 이상, 필드로 삽입하는 `조합`

### 효과
- `독립성 확보` 상속/재정의로 인해 발생하는 부모 클래스의 메서드 오염을 방지
- `유연성 확보` 인스턴스 주입 방식으로 다형성 유연성 확장성? 확보

### 피드백 해석
> `공통 개념으로 묶은 것이 행위인지, 아니면 단순히 핸드와 버스트 등의 상태인지`

- 행위 `IS-A`: "딜러와 플레이어는 둘 다 게임의 참가자로서 카드를 받을 수 있다"  
  해당 클래스들의 공통된 행위로 묶은, 다형성(Polymorphism)을 위한 올바른 설계
- 상태 `HAS-A`: "딜러와 플레이어는 둘 다 가드 뭉치를 가지고 있고 그 점수를 계산할 수 있다"  
  해당 클래스들의 공통된 행위가 아닌, 해당 클래스들의 공통된 상태의 행위를 기준으로 묶은 상황  
  + 딜러가 수익률 판별을 진행, `LSP` 리스코프 치환 원칙 위반

### 피드백 답변
> 특정 상태와 행위를 공유하고,  
> 그 행위의 세부가 다르기에 오버라이딩하는 방식으로 구현했었습니다.  
다만 아루의 질문에 대한 답을 생각해 보면 이전 `PlayingCards` 의 사례처럼  
일부분적인 동일성(카드 뭉치를 관리한다)에 몰두해  
묶여 있을 필요가 없던 `Deck` 과 `Hand` 를 묶어버렸듯이  
일부분적인 동일성(손패를 상태로 가진다)에 몰두해  
묶여 있을 필요가 없는 `Dealer` 와 `Player` 가 묶여 있는 것이 아닌가 싶네요.
>
> 아! 이전의 질문 `중복된 코드를 제거한다 의 방법이 상속 뿐일까요?` 의 답이  
여기에서 나올 수 있는 것 같아요. 결국 카드를 받는다 / 버스트 여부를 확인한다 같은  
중복된 코드를 `Hand` 라는 도메인으로 묶어 딜러와 플레이어에게 준다면  
필요한 기능만을 추출해낼 수 있기 때문이에요.
>

## 리뷰어 피드백

    질문을 통해 스스로 답을 도출하도록 유도하고,
    그 과정에서 자율적 학습 방식과 발전 경험을 습득할 수 있도록 지도.
    
    초반 피드백에선 핵심 키워드와 방향성을 보다 명시적으로 제시하는 형식으로
    혼동이나 오해를 차단하고 바람직한 답을 찾을 수 있도록 유도하며,
    
    언급되었던 주제나 개념에 중복되는 내용, 보완 사항은
    해당 부분을 특정하고, 생각과 의도에 대한 질문을 통해
    리뷰이의 의견이나 고민을 청취하고 그 과정에서 스스로 내용을 환기하도록 함.
    
    개념과 기준이 명확하지 않은 입장에서 헷갈리거나 빼먹기 쉬운 부분을
    직접 정립하고 정리할 수 있도록 유도함으로써
    경험적 학습을 통해 스스로의 생각과 방향성을 수립하는 데,
    정확히는 수립을 위한 기본 토대를 쌓는 데 매우 큰 도움이 되었다.
    
    한 번에 이해하고 적용하기 어려운 개념이나 방식을 설명할 때,
    그와 연관된 여러 부분을 각각 짚어 주고
    그 부분적 개선과 변화가 집적되어
    하나의 원칙이나 개념이 성립되는 모습을 직접 경험하게끔 하고,
    
    또한 역으로 리팩토링된 하나의 바람직한 구조가 얼마나 다양한 문제를 해결하고
    앞으로의 문제를 방지할 수 있는지 경험시켜 준 부분이 큰 도움이 되었다.
    
    가장 만족스럽고 가장 감사했던 부분은 반복적인 피드백에 대한 적극적인 대응.
    많은 시간과 노력을 투입해 상대적으로 빠른 리뷰와 피드백을 제공하고,
    예상되는 완료 시간을 고지함으로써
    리뷰이의 체계적인 시간 관리와 일정 조율을 가능하게 해 주었다.
    
    빠른 PR 에 빠른 RC 로 화답하며 많은 소통과 개선을 시도할 수 있었다는 점이
    리뷰어/리뷰이 관계의 학습 방식을 매우 효율적으로 활용한 부분이라 생각.
