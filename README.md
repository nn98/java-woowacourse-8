# ✅ 나만의 원칙

> 코딩 전반에 있어 나만의 원칙과 규칙, 메뉴얼과 가이드를 준수.

## 주요 원칙

1. ### 코드 스타일, 가독성
- 불필요한 주석 지양. 코드만으로 파악할 수 없는 배경이나 맥락에 대한 설명만을 남긴다.
- 시그니처, 인스턴스, 필드의 이름도 유의미하게. 이름만으로도 내용을 추론할 수 있어야 한다.
- 컨벤션 준수, 포매터 적용, 3항 연산자 배제, else 예약어 배제, 들여쓰기 최소화.
- 메소드 파라미터가 한계에 근접한다면 줄바꿈으로 가독성 확보. _**※ Mind_Bridge**_ 리팩토링 과정에서 추가

2. ### 클래스, 메서드
- 작성 순서는 상수 > 멤버 변수 > 생성자 > public 메서드 > private 메서드.
- 메서드는 단일 책임 원칙 준수. 중복 코드는 분리ㆍ추출, 15줄 이내로 정리.
- 메서드의 인자, 클래스의 인스턴스 변수를 최소화.
- 메서드 인자는 0~3개를 목표로, 초과시 객체/DTO로 묶기 시도.
- 클래스의 인스턴스 변수는 3개 이하를 목표로, 초과시 분리/추출 시도.
- 정적 상수(private static final)은 인스턴수 변수라기보단 포장된 원시값으로 판단. _**※ Mind_Bridge**_
- 테스트 등 코드의 목적이 다를 경우 원칙보단 근본에 집중해 과감하게 활용. _**※ Mind_Bridge**_
- 정적 상수는 헬퍼 메서드에서 한 번에 참조, 테스트는 헬퍼 메서드를 참조. _**※ Mind_Bridge**_
- 계산 가능한 값은 필드`데이터`로 저장하기보단 계산`행위`하기.
- 위와 더불어 직접적인 getter / setter + 값 조작 대신 그 일을 해당 클래스에 위임하기.
- 불변 가능한 값은 불변화, 원시값은 포장, 일급 콜렉션 활용.
- 기능 > 객체 > 메서드. 기능은 객체의 조합, 객체는 메서드의 조합, 메서드는 하나의 동작을 담당.
- 상수의 집합화가 가능하다면 enum 으로 static final 값을 묶어 표현. 타입 안정성과 가독성 보장.

3. ### 계층화, 패턴화
- 비즈니스 로직 / UI 로직 분리. 각 계층은 자신의 역할에만 집중하고, 단방향 의존과 배타성을 보장한다.
- [제리의 MVC 패턴] - MVC 5원칙 준수.
- 모델은 컨트롤러에도 뷰에도 의존하지 않는다.
- 뷰는 모델에만 의존하고, 컨트롤러에 의존하지 않는다.
- 컨트롤러는 모델에도 뷰에도 의존할 수 있다.
- 뷰가 모델에서 받는 데이터는 공통값이 아닌 특정한, 고유한 값에 한정한다.
- 뷰가 모델에서 받는 데이터는 컨트롤러를 거쳐(컨트롤러를 통해)야 한다.

4. ### 테스트
- 작은 단위 테스트부터 점진적 테스트 확장.
- 테스트 메서드도 메서드와 같이 단일 책임과 중복 추출, 다양한 도구와 API 활용.
- 성공/예외/엣지 케이스 및 빈(empty)값, 멱등 등 다양한 경우의 수를 복합적으로 테스트. _**※ Mind_Bridge**_
- 반복되는 필드 검증, DTO 비교 등은 분리/추출. _**※ Mind_Bridge**_

5. ### 리팩토링
- 리팩토링을 위해 정상 동작 여부를 파악. 테스트 필수.
- 각 메서드/클래스에 대한 테스트 먼저 구현
- 하나의 책임/기능/동작 단위로 리팩토링 진행. 단일 책임 원칙 준수.
- 기능이 혼재된 경우 기능 분리를 선결
- 커밋 하나 – 기능 하나 관계를 보장, 변경과 개선 명확화
- 각 원칙을 기반으로 작은 단위부터 점진적으로 리팩토링

**원칙, 규칙. 클린코드, TDA, SRP, LOD, DRY**

## 작업 가이드ㆍ메뉴얼ㆍ규칙

1. ### `테스트는 쉬운 부분부터.`  
   순수 테스트 구현이 쉬운 부분부터 어려운 부분까지 점진적으로 테스트 구현. <br><br>

1. ### `테스트는 작은 부분부터.`   
   단위 테스트부터 통합 테스트, 시스템/인수 테스트 등 크고 복잡한 테스트까지. <br><br>

1. ### `테스트와 비즈니스 로직의 상호 보완.`   
   선후와 관계 없이, 로직의 유효한 동작을 테스트로 보장할 수 있도록 구현. <br><br>

1. ### `근본적 목적을 기억하기.`   
   문제 상황 해결과 정상적 동작이라는 핵심 목표를 달성한 뒤, 원칙과 규칙을 적용해 리팩토링. <br><br>

1. ### `한 번에 한 가지씩.`  
   like SRP. 한 번에 모든 원칙을 도입하려 하지 말고 특정 원칙을 특정한 범위에만.  <br><br>

1. ### `테스트가 보장해주는 범위를.`   
   정상 동작을 반드시 테스트로 보장하고 그 아래에서 원칙을 적용해 리팩토링. <br><br>

1. ### `리팩토링은 핵심적인 부분부터.`   
   단순한 로직, 말단부는 변경 가능성이 높기에 후순위로. <br><br>

1. ### `리팩토링은 핵심적인 원칙부터.`   
   스타일 같은 간단한 원칙은 후순위로 통합 적용. <br><br>
